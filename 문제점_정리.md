# miniDBMS 문제점 정리

작성일: 2026-04-22

이 문서는 저장소 전체를 정적 검토하면서 발견한 문제를 정리한 것이다.
현재 작업 환경에서는 `make`, `gcc`가 Windows PATH에 없고 WSL도 `/bin/bash`가 없어 실제 빌드/테스트 실행은 확인하지 못했다.

## 검토 범위

- 빌드/컨테이너 설정: `.devcontainer/*`, `Makefile`, `.gitignore`, `.gitattributes`
- 공용 계약: `include/*.h`
- 엔진 소스: `src/input`, `src/parser`, `src/schema`, `src/executor`, `src/index`, `src/bptree`
- 서버/API 소스: `src/http`, `src/service`, `src/server`, `src/threadpool`
- 테스트/샘플: `tests/*`, `samples/*`, `schema/*`, `data/*`
- 문서: 루트 Markdown, `docs/*`

## 요약

| 우선순위 | 문제 | 영향 |
|---|---|---|
| P1 | devcontainer가 실제 Dockerfile을 찾지 못할 수 있음 | 컨테이너 개발 환경 빌드 실패 |

## 1. devcontainer Dockerfile 이름 불일치

- 우선순위: P1
- 위치:
  - `.devcontainer/devcontainer.json:3`
  - `.devcontainer/dockerfile`

### 현재 상태

`devcontainer.json`은 아래처럼 대문자 `Dockerfile`을 참조한다.

```json
"dockerFile": "Dockerfile"
```

하지만 실제 파일명은 `.devcontainer/dockerfile`로 소문자다.

### 왜 문제인가

Windows에서는 파일명 대소문자가 느슨하게 처리될 수 있지만, Linux/컨테이너 환경에서는 `Dockerfile`과 `dockerfile`이 다른 파일이다.
따라서 devcontainer 빌드가 실제 환경에서 실패할 수 있다.

### 수정 방향

둘 중 하나로 통일한다.

1. `.devcontainer/dockerfile`을 `.devcontainer/Dockerfile`로 rename
2. `.devcontainer/devcontainer.json`의 값을 `"dockerfile"`로 변경

일반적으로 Docker 관례에 맞춰 파일명을 `Dockerfile`로 바꾸는 편이 낫다.

### 확인 방법

```bash
devcontainer build .
```

또는 VS Code에서 Dev Container reopen을 시도해서 이미지 빌드가 정상인지 확인한다.

## 2. service API multi-statement 차단 불완전

- 우선순위: P1
- 위치: `src/service/db_service.c:134-143`

### 현재 상태

`db_service_execute_sql()`은 MVP 기준으로 한 요청에 하나의 SQL statement만 허용한다고 되어 있다.
하지만 현재 구현은 세미콜론 개수만 세서 multi-statement 여부를 판단한다.

```c
if (semi_count > 1) {
    ...
    "multi-statement not supported"
}
```

### 재현 가능한 입력

```sql
SELECT * FROM users; INSERT INTO users VALUES (10, 'x', 20, 'x@test.com')
```

위 입력은 statement가 2개지만 세미콜론은 1개뿐이다.
현재 로직에서는 `semi_count > 1`이 아니므로 거절되지 않는다.

### 왜 문제인가

`extract_first_statement()`는 첫 번째 statement만 잘라서 parser에 넘긴다.
따라서 위 입력은 두 번째 `INSERT`가 실행되지 않고 조용히 무시될 가능성이 크다.

이 동작은 API 사용자 입장에서 매우 위험하다.
요청은 성공처럼 보이는데 일부 SQL이 처리되지 않기 때문이다.

### 수정 방향

세미콜론 개수만 보지 말고, 첫 statement 이후 EOF 전까지 의미 있는 토큰이 남아 있는지 검사해야 한다.

예시 방향:

1. 첫 statement 종료 위치를 찾는다.
2. 종료 위치가 세미콜론이면 그 다음 토큰으로 이동한다.
3. 다음 토큰이 `TOKEN_EOF`가 아니면 `DB_SERVICE_BAD_REQUEST`를 반환한다.

### 추가 테스트

`tests/test_executor.c` 또는 별도 service 테스트에 아래 케이스를 추가한다.

```c
db_service_execute_sql(
    "SELECT * FROM users; INSERT INTO users VALUES (10, 'x', 20, 'x@test.com')",
    &opts,
    &res
);
```

기대값:

- `DB_SERVICE_BAD_REQUEST`
- 메시지: multi-statement not supported 계열

## 3. 컬럼 지정 INSERT에서 필수 컬럼 누락 허용

- 우선순위: P1
- 위치:
  - `src/schema/schema.c:167-225`
  - `src/executor/executor.c:630-640`

### 현재 상태

컬럼 지정 INSERT는 지정한 컬럼명과 값만 검증한다.

예:

```sql
INSERT INTO users (id, name) VALUES (3, 'kim');
```

현재 `schema/users.schema`는 다음과 같다.

```text
col0=id,INT,0
col1=name,VARCHAR,64
col2=age,INT,0
col3=email,VARCHAR,128
```

하지만 `age`, `email`이 빠져도 schema validation 단계에서 막지 않는다.

이후 executor는 누락된 컬럼을 빈 문자열로 채운다.

```c
fprintf(fp, "%s", val_idx >= 0 ? stmt->values[val_idx] : "");
```

### 왜 문제인가

`age`는 `INT` 컬럼인데 빈 문자열로 저장될 수 있다.
그 결과 data 파일에는 스키마상 유효하지 않은 row가 들어간다.

또한 `db_insert()`는 age 값이 없으면 age 인덱스를 갱신하지 않는다.
즉 원본 data 파일과 메모리 인덱스가 서로 달라진다.

### 영향

- `SELECT *`에서는 row가 보이지만 `WHERE age BETWEEN ...`에서는 누락될 수 있음
- data 파일에 스키마와 맞지 않는 row가 저장됨
- API 서버 모드에서는 장시간 실행 중 인덱스 불일치가 누적될 수 있음

### 수정 방향

정책을 먼저 정해야 한다.

1. 가장 단순한 정책: 컬럼 지정 INSERT도 모든 컬럼을 반드시 제공하게 한다.
2. 확장 정책: schema에 nullable/default 정보를 추가하고 누락 컬럼을 명시적으로 처리한다.

현재 schema 형식에는 nullable/default 정보가 없으므로 1번이 안전하다.

구현 방향:

- `schema_validate()`에서 `ins->column_count > 0`인 경우에도 schema의 모든 컬럼이 제공됐는지 확인
- 중복 컬럼명도 거절
- 값 개수와 컬럼 개수 일치 여부는 기존처럼 유지

### 추가 테스트

```sql
INSERT INTO users (id, name) VALUES (1, 'alice');
```

기대값:

- `SQL_ERR`

```sql
INSERT INTO users (id, name, age, email) VALUES (1, 'alice', 20, 'a@test.com');
```

기대값:

- `SQL_OK`

## 4. index_init이 고정 컬럼 위치를 가정

- 우선순위: P2
- 위치: `src/index/index_manager.c:104-110`

### 현재 상태

인덱스 초기화는 data 파일의 0번 컬럼을 id, 2번 컬럼을 age로 가정한다.

```c
col_value(line, 0, col_buf, sizeof(col_buf)); // id
col_value(line, 2, col_buf, sizeof(col_buf)); // age
```

### 왜 문제인가

현재 `users`와 `test_exec_tmp` 스키마에서는 우연히 맞는다.
하지만 아래처럼 컬럼 순서가 바뀌면 인덱스가 틀어진다.

```text
col0=name,VARCHAR,64
col1=id,INT,0
col2=email,VARCHAR,128
col3=age,INT,0
```

이 경우 id 인덱스에는 name을 `atoi()`한 값이 들어가고, age 인덱스에는 email을 `atoi()`한 값이 들어갈 수 있다.

### 영향

- `WHERE id = ...` 조회 실패
- `WHERE id BETWEEN ...` 조회 실패
- `WHERE age BETWEEN ...` 조회 실패
- linear scan 결과와 인덱스 결과 불일치

### 수정 방향

index 계층이 table schema를 알 수 있게 만들어야 한다.

가능한 방법:

1. `index_init()`에서 `schema_load(table)`을 호출해 id/age 컬럼 인덱스를 찾는다.
2. 또는 executor/service가 이미 로드한 `TableSchema`를 index 계층에 넘기는 API를 추가한다.

역할 분리 관점에서는 2번이 더 명시적이지만 공용 API 변경이 필요하다.
빠른 수정은 1번이다.

### 추가 테스트

컬럼 순서가 다른 임시 스키마를 만들고 다음을 비교한다.

- 인덱스 경로 `SELECT * FROM table WHERE id = 3`
- 강제 linear 경로 `SELECT * FROM table WHERE id = 3`

두 결과 row_count와 row 값이 같아야 한다.

## 5. id 중복 시 point lookup 결과 누락 가능

- 우선순위: P2
- 위치:
  - `src/bptree/bptree.c:646-664`
  - `src/executor/executor.c:517-521`

### 현재 상태

B+Tree는 같은 key에 여러 offset을 저장할 수 있다.
하지만 `bptree_search()`는 첫 번째 offset 하나만 반환한다.

```c
return leaf->values[pos]->offsets[0];
```

executor의 id equality 경로도 이 offset 하나만 읽는다.

```c
long offset = index_search_id(stmt->table, id);
rs = fetch_by_offset(offset, stmt, schema);
```

### 왜 문제인가

현재 schema에는 primary key나 unique 제약이 없다.
따라서 같은 id를 가진 row가 여러 개 들어갈 수 있다.

그 상태에서:

```sql
SELECT * FROM users WHERE id = 1;
```

인덱스 경로는 첫 번째 row만 반환하고, linear scan은 id가 1인 모든 row를 반환한다.

### 수정 방향

둘 중 하나를 선택해야 한다.

1. id를 unique로 강제한다.
   - `db_insert()` 전에 `index_search_id()`로 기존 id 존재 여부를 확인하고 중복이면 실패
   - schema 문서에 id unique 규칙 명시

2. id equality도 여러 offset을 반환하게 한다.
   - `index_search_id_alloc()` 같은 API 추가
   - 또는 `id BETWEEN x AND x`와 같은 range API를 재사용
   - executor는 `fetch_by_offsets()`로 모든 row를 반환

DBMS 관점에서는 id를 primary key처럼 볼 거라면 1번이 단순하다.
테이블이 일반 row store라면 2번이 정확하다.

### 추가 테스트

```sql
INSERT INTO users VALUES (1, 'a', 20, 'a@test.com');
INSERT INTO users VALUES (1, 'b', 30, 'b@test.com');
SELECT * FROM users WHERE id = 1;
```

정책에 따라 기대값을 정한다.

- unique 정책: 두 번째 INSERT 실패
- non-unique 정책: SELECT 결과 2행

## 6. 샘플 INSERT가 users 스키마와 불일치

- 우선순위: P2
- 위치:
  - `samples/insert.sql:1`
  - `schema/users.schema:1-6`

### 현재 상태

`schema/users.schema`:

```text
columns=4
col0=id,INT,0
col1=name,VARCHAR,64
col2=age,INT,0
col3=email,VARCHAR,128
```

`samples/insert.sql`:

```sql
INSERT INTO users VALUES (13, '김세민', 'male', T);
```

세 번째 값 `'male'`은 `age INT` 위치에 들어가고, 네 번째 값 `T`는 `email VARCHAR` 위치에 들어간다.

### 왜 문제인가

schema validation에서 `age`가 INT가 아니므로 실패해야 정상이다.
즉 기본 샘플 파일이 현재 코드/스키마 기준으로는 성공 예제가 아니다.

### 수정 방향

샘플을 현재 스키마에 맞춘다.

예:

```sql
INSERT INTO users VALUES (13, '김세민', 25, 'kim@example.com');
SELECT * FROM users;
```

또는 gender/active 컬럼을 쓰고 싶다면 `schema/users.schema`를 바꿔야 한다.
다만 현재 `tools/gen_data.c`, 테스트, 인덱스 코드는 `id, name, age, email` 순서를 기준으로 작성되어 있으므로 샘플을 고치는 편이 안전하다.

## 9. 빌드 산출물 test_http가 Git에 추적 중

- 우선순위: P3
- 위치:
  - `test_http`
  - `.gitignore`

### 현재 상태

루트의 `test_http`는 ELF 바이너리다.

확인된 헤더:

```text
7F 45 4C 46
```

`.gitignore`에는 여러 테스트 바이너리가 등록되어 있지만 `test_http`가 없다.
그리고 현재 `test_http`는 Git에 추적 중이다.

### 왜 문제인가

- 플랫폼 의존 바이너리가 저장소에 들어간다.
- 소스와 무관한 바이너리 변경이 diff에 잡힐 수 있다.
- 다른 OS 사용자가 실행할 수 없는 파일이 저장소에 남는다.

### 수정 방향

1. `.gitignore`에 `test_http` 추가
2. 추적 중인 바이너리 제거

```bash
git rm --cached test_http
```

주의: 파일 자체를 삭제할지, Git 추적만 끊을지는 팀 정책에 따라 결정한다.

## 10. 테스트 실행 환경 문제

- 우선순위: P3
- 위치: 로컬 개발 환경

### 현재 상태

현재 PowerShell 환경에서 다음 명령을 찾지 못했다.

```text
make
gcc
```

WSL 실행도 실패했다.

처음에는 권한 문제로 `E_ACCESS_DENIED`가 발생했고, 권한을 올린 뒤에는 `/bin/bash`가 없다는 오류가 발생했다.

### 영향

정적 리뷰는 가능하지만 실제 컴파일 경고, 링크 오류, 테스트 실패 여부는 확인할 수 없다.

### 수정 방향

다음 중 하나를 준비해야 한다.

1. devcontainer 정상화 후 컨테이너에서 테스트
2. Windows에 MSYS2/MinGW 또는 Make/GCC 설치
3. WSL 배포판 설치 및 `/bin/bash` 사용 가능 상태 확인

### 확인 명령

```bash
make clean
make test
```

가능하면 다음도 추가로 실행한다.

```bash
bash scripts/lint.sh
bash scripts/check_binary_mode.sh
```

4. 컬럼 지정 INSERT의 필수 컬럼 누락/중복 컬럼 검증 추가
5. id 중복 정책 결정 후 executor/index/B+Tree 경로 수정
6. `index_init()`이 schema 기반으로 컬럼 위치를 찾도록 수정
7. README 정리 및 깨진 이미지 링크 처리
8. `test_http` 추적 제거 및 `.gitignore` 보강
9. 정상 빌드 환경에서 `make test` 실행

## 최소 회귀 테스트 목록

아래 케이스는 수정 후 반드시 확인하는 것이 좋다.

```sql
-- 정상 INSERT
INSERT INTO users VALUES (1, 'alice', 20, 'alice@example.com');

-- 컬럼 지정 정상 INSERT
INSERT INTO users (id, name, age, email) VALUES (2, 'bob', 30, 'bob@example.com');

-- 컬럼 누락 INSERT: 실패해야 함
INSERT INTO users (id, name) VALUES (3, 'carol');

-- id point lookup
SELECT * FROM users WHERE id = 1;

-- id range lookup
SELECT * FROM users WHERE id BETWEEN 1 AND 10;

-- age range lookup
SELECT * FROM users WHERE age BETWEEN 20 AND 30;

-- linear fallback
SELECT * FROM users WHERE name = 'alice';
```

API/service 경로:

```text
POST /query
{"sql":"SELECT * FROM users; INSERT INTO users VALUES (10, 'x', 20, 'x@test.com')"}
```

기대값:

- HTTP 400
- multi-statement not supported 계열 오류
