# 테스트 방법

이 문서는 현재 저장소 상태 기준으로 **수요코딩회 발표에서 실제로 볼 테스트**만 정리한다.

목표는 단순하다.

> "우리 miniDBMS가 API 요청을 해석하고, 기존 SQL 엔진을 호출하고, thread pool로 여러 요청을 처리한다"를 확인한다.

이전 차수용 B+Tree, index, parser, schema 테스트는 현재 작업트리에서 테스트 소스가 사라진 상태라 이 문서의 실행 대상으로 보지 않는다.

---

## 1. 현재 남아 있는 테스트

현재 `tests/` 폴더에 실제로 남아 있는 테스트 소스는 3개다.

| 파일                      | 확인하는 것                                       | 수요코딩회와의 관계              |
| ------------------------- | ------------------------------------------------- | -------------------------------- |
| `tests/test_http.c`       | HTTP 요청 파싱, 오류 메타데이터, JSON 응답 직렬화 | API 요청/응답 모양 검증          |
| `tests/test_executor.c`   | `db_service_execute_sql`로 SQL 엔진 통합 실행     | API 서버가 호출할 엔진 경계 검증 |
| `tests/test_threadpool.c` | thread pool job 병렬 처리와 worker 수별 시간 비교 | 동시 요청 병렬 처리 근거         |

쉽게 말하면:

```text
test_http.c
  "서버가 받을 HTTP 요청/응답 형식을 이해하는가?"

test_executor.c
  "SQL 문자열이 service 경계를 통해 실제 엔진까지 실행되는가?"

test_threadpool.c
  "여러 SQL job을 worker들이 병렬로 처리하는가?"
```

---

## 2. 현재 남아 있는 항목

현재 남아 있는 thread pool 외부 요청 스크립트는 이것이다.

```text
scripts/threadpool_smoke.ps1
```

요청 수를 바꾸고 싶으면 별도 스크립트가 아니라 `-ConcurrentRequests` 옵션을 사용한다.

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\threadpool_smoke.ps1 -ConcurrentRequests 16
powershell -ExecutionPolicy Bypass -File .\scripts\threadpool_smoke.ps1 -ConcurrentRequests 32
powershell -ExecutionPolicy Bypass -File .\scripts\threadpool_smoke.ps1 -ConcurrentRequests 64
powershell -ExecutionPolicy Bypass -File .\scripts\threadpool_smoke.ps1 -ConcurrentRequests 128
```

## 3. Makefile 주의

현재 `Makefile`의 `test` target은 아직 `test_parser`, `test_schema`를 포함한다.

```make
TEST_BINS = test_parser test_schema test_executor test_http test_threadpool
```

하지만 `tests/test_parser.c`, `tests/test_schema.c`가 현재 없기 때문에, 이 상태에서는 `make test` 또는 `mingw32-make test`가 그대로 성공한다고 기대하면 안 된다.

현재는 아래처럼 **남아 있는 테스트 target을 개별 실행**하는 쪽이 안전하다.

```bash
make test_http
./test_http

make test_executor
./test_executor

make test_threadpool
./test_threadpool
```

Windows PowerShell에서는 실행 파일 확장자가 붙을 수 있다.

```powershell
mingw32-make test_http
.\test_http.exe

mingw32-make test_executor
.\test_executor.exe

mingw32-make test_threadpool
.\test_threadpool.exe
```

## 4. 테스트 1: HTTP 메시지 단위 테스트

이 테스트는 서버를 띄우지 않는다.

실행:

```bash
make test_http
./test_http
```

Windows:

```powershell
mingw32-make test_http
.\test_http.exe
```

확인하는 것:

- `POST /query` 요청 body에서 SQL과 옵션을 꺼낼 수 있는가
- 잘못된 route를 `404` 성격의 오류로 처리하는가
- 잘못된 boolean option을 `400` 성격의 오류로 처리하는가
- 공백뿐인 SQL을 거절하는가
- 잘못된 `request_id` 타입을 거절하는가
- 파싱 실패 시에도 `request_id`를 응답 메타데이터에 보존하는가
- `DBServiceResult`를 JSON 응답 문자열로 직렬화하는가

중요:

> `test_http.c`는 실제 HTTP 서버를 켜지 않는다.  
> socket, port, curl 요청을 확인하는 테스트가 아니라 `src/http/http_message.c`의 파싱/직렬화 함수만 직접 호출하는 단위 테스트다.

---

## 5. 테스트 2: service + executor 통합 테스트

실행:

```bash
make test_executor
./test_executor
```

Windows:

```powershell
mingw32-make test_executor
.\test_executor.exe
```

확인하는 것:

- `INSERT`가 정상 실행되는가
- `SELECT *`가 정상 실행되는가
- `WHERE id = N` 조회가 되는가
- `WHERE id BETWEEN a AND b` 조회가 되는가
- `WHERE age BETWEEN a AND b` 조회가 되는가
- 인덱스 경로와 linear scan 경로의 결과가 일치하는가
- `db_service_execute_sql`로 SQL 문자열을 넣었을 때 parser/schema/executor까지 연결되는가
- invalid SQL, empty SQL, compare 미지원 옵션을 안전하게 처리하는가

수요코딩회 관점에서 가장 중요한 부분은 `db_service_execute_sql` 통합 경로다.

```text
SQL string
  -> db_service_execute_sql()
  -> lexer/parser
  -> schema validation
  -> executor
  -> ResultSet or rows_affected
```

즉, 이 테스트는 API 서버가 실제로 호출할 DB 엔진 경계가 동작하는지 확인한다.

## 6. 테스트 3: thread pool 내부 병렬 처리 테스트

실행:

```bash
make test_threadpool
./test_threadpool
```

Windows:

```powershell
mingw32-make test_threadpool
.\test_threadpool.exe
```

이 테스트가 하는 일:

1. 테스트용 테이블 `tp_smoke`를 만든다.
2. 여러 `SELECT` job을 thread pool에 넣는다.
3. worker 수를 `1`, `2`, `4`로 바꿔 같은 작업을 실행한다.
4. 4개 worker가 1개 worker보다 확실히 빠른지 확인한다.
5. 완료 시간이 worker 수만큼 wave처럼 묶여 나오는지 보여준다.

중요:

> 이 테스트는 실제 TCP 서버를 켜는 테스트가 아니다.  
> thread pool 내부에 SQL job을 직접 넣어서 병렬 처리 근거를 확인하는 C 테스트다.

발표에서는 이렇게 말하면 된다.

```text
외부 API 테스트와 별도로, 내부 thread pool 테스트에서 worker 수를 1개, 2개, 4개로 바꿔 같은 SQL job 묶음을 실행했습니다.
4개 worker가 더 짧은 시간에 끝나서 병렬 처리 구조가 실제로 동작함을 확인했습니다.
```

## 7. 실제 API 서버 실행

이 단계는 Linux 또는 WSL에서 하는 것이 안전하다.

Windows 빌드에서는 실제 socket server가 지원되지 않을 수 있다. Windows에서 아래 명령을 실행했을 때:

```powershell
.\sqlp.exe --server 8080
```

다음 메시지가 나오면 현재 Windows 빌드에서 서버 모드가 비활성화된 것이다.

```text
[server] --server mode is not supported in this Windows build yet.
```

Linux/WSL에서 빌드:

```bash
make all
```

서버 실행:

```bash
./sqlp --server 8080 --threads 4
```

정상 로그 예시:

```text
[server] miniDBMS HTTP server listening on port 8080 with 4 worker thread(s)
```

이 터미널은 끄지 말고 그대로 둔다.

## 8. 서버 health 확인

새 터미널에서 실행한다.

```bash
curl -i http://127.0.0.1:8080/health
```

성공 기준:

- HTTP 상태 코드가 `200`
- 응답에 `"ok":true`
- 응답에 `"message":"healthy"` 또는 health 성공을 의미하는 JSON이 있음

이 단계는 `test_http.c`와 다르게 실제 서버에 네트워크 요청을 보내는 확인이다.

---

## 9. SQL을 API로 보내기

### INSERT 요청

```bash
curl -i -X POST http://127.0.0.1:8080/query \
  -H "Content-Type: application/json" \
  -d "{\"sql\":\"INSERT INTO users VALUES (1, 'alice', 30, 'alice@example.com');\"}"
```

성공 기준:

- HTTP 상태 코드가 `200`
- `"ok":true`
- `"rows_affected":1`

### SELECT 요청

```bash
curl -i -X POST http://127.0.0.1:8080/query \
  -H "Content-Type: application/json" \
  -d "{\"sql\":\"SELECT * FROM users WHERE id = 1;\",\"include_profile\":true}"
```

성공 기준:

- HTTP 상태 코드가 `200`
- `"ok":true`
- `"row_count":1`
- 응답 rows 안에 `alice`가 있음
- `"profile"`이 있으면 실행 경로도 확인 가능

## 10. Postman으로 확인하기

발표 때는 Postman이 보기 쉽다.

사용할 파일:

```text
postman/miniDBMS.postman_collection.json
```

순서:

1. Linux/WSL에서 API 서버를 먼저 켠다.
2. Postman을 연다.
3. `postman/miniDBMS.postman_collection.json` 파일을 import 한다.
4. `base_url`이 `http://127.0.0.1:8080`인지 확인한다.
5. Health, Insert, Select 계열 요청을 순서대로 실행한다.

Postman은 실제 API 서버에 요청을 보내는 테스트다. 따라서 `test_http.c`보다 발표 시연에 더 직접적이다.

## 11. 외부 동시 요청 smoke 테스트

서버가 켜진 상태에서 Windows PowerShell을 열고 실행한다.

프로젝트 루트에 있을 때:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\threadpool_smoke.ps1
```

요청 수를 늘리고 싶으면:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\threadpool_smoke.ps1 -ConcurrentRequests 16
powershell -ExecutionPolicy Bypass -File .\scripts\threadpool_smoke.ps1 -ConcurrentRequests 32
```

이 스크립트가 하는 일:

1. `/health`로 서버가 살아있는지 확인한다.
2. 테스트용 row를 여러 개 INSERT 한다.
3. 여러 `SELECT` 요청을 동시에 보낸다.
4. 모든 요청이 성공했는지 확인한다.

성공하면 마지막에 다음 의미의 결과가 나온다.

```text
Thread pool smoke test passed.
```

만약 연결 실패가 나오면 스크립트 문제가 아니라 API 서버가 켜져 있지 않거나 응답하지 않는 상태일 가능성이 높다.

## 12. Worker 로그로 병렬 처리 보기

서버를 trace 모드로 켠다.

```bash
MINIDBMS_TRACE_WORKERS=1 ./sqlp --server 8080 --threads 4
```

요청마다 일부러 지연을 넣어 worker들이 동시에 일하는 모습을 더 잘 보고 싶으면:

```bash
MINIDBMS_TRACE_WORKERS=1 MINIDBMS_WORKER_DELAY_MS=500 ./sqlp --server 8080 --threads 4
```

그 다음 PowerShell에서 동시 요청 smoke 테스트를 실행한다.

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\threadpool_smoke.ps1 -ConcurrentRequests 8
```

서버 터미널에 여러 worker의 `start`, `done` 로그가 섞여 보이면 여러 worker가 요청을 나눠 처리한 것이다.

## 13. 발표용 정리

발표에서는 테스트를 이렇게 묶어 말하면 된다.

```text
첫째, test_http로 API 요청/응답 메시지 형식을 단위 테스트했습니다.
둘째, test_executor로 API 서버가 호출할 db_service 실행 경계를 검증했습니다.
셋째, test_threadpool로 worker 수가 늘 때 같은 SQL job 묶음이 더 빨리 끝나는지 확인했습니다.
넷째, 실제 서버를 켠 뒤 curl/Postman/threadpool_smoke.ps1로 외부 클라이언트 요청이 처리되는지 확인했습니다.
```

더 짧게 말하면:

```text
HTTP 메시지, DB 엔진 호출 경계, thread pool 병렬 처리, 실제 외부 API 요청을 나눠 검증했습니다.
```

아래 항목은 현재 파일이 없거나 수요코딩회 발표 핵심에서 벗어나므로 자세한 실행 절차에서 뺐다.

- `test_bptree.c`
- `test_index.c`
- `test_parser.c`
- `test_schema.c`
- `threadpool_16.ps1`
- `threadpool_32.ps1`
- `threadpool_64.ps1`
- `threadpool_128.ps1`

이전 차수 엔진 검증이 다시 필요하면 해당 테스트 소스를 복구한 뒤 Makefile의 `test` target도 함께 정리해야 한다.
