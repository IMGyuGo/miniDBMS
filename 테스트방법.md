# 테스트 방법

이 문서는 **수요코딩회 발표에서 보여줄 테스트만** 정리한 문서다.

목표는 단순하다.

> "우리 미니 DBMS가 API 서버로 켜지고, 외부 요청을 받고, SQL을 실행하고, 여러 요청도 처리한다"를 확인한다.

그래서 이전 차수 확인용인 B+Tree, parser, schema 같은 세부 테스트 설명은 여기서 뺐다.

---

## 1. 이번에 확인할 것

수요코딩회 요구사항 기준으로 기본 시연 테스트는 4개다.
여기에 병렬 처리 근거를 더 잘 설명하기 위한 추가 테스트 2개를 더 준비했다.

| 번호 | 테스트 | 왜 하나요? |
|---|---|---|
| 1 | HTTP 단위 테스트 | `/query`, `/health` 요청/응답 모양이 맞는지 확인 |
| 2 | API 서버 실행 테스트 | 서버가 진짜 켜지는지 확인 |
| 3 | Postman API 테스트 | 사람이 직접 API를 눌러서 INSERT/SELECT 확인 |
| 4 | Thread Pool 동시 요청 테스트 | 여러 요청을 동시에 보내도 처리되는지 확인 |
| 5 | 16/32/64/128 요청 스케일 테스트 | 동시 요청 수를 늘릴 때 처리량이 어디서 좋아지는지 확인 |
| 6 | Worker trace 테스트 | 여러 worker 스레드가 요청을 나눠 받는지 로그로 확인 |
| 7 | 1/2/4 worker 시간 비교 | worker 수가 늘면 같은 작업이 더 빨라지는지 확인 |

쉽게 말하면:

1. 코드가 HTTP 요청을 알아듣는지 본다.
2. 서버를 켠다.
3. Postman으로 SQL을 보내본다.
4. 여러 요청을 한꺼번에 보내본다.
5. 16개, 32개, 64개, 128개 요청을 차례대로 보내본다.
6. 서버 로그에서 여러 worker가 움직이는지 본다.
7. worker 수를 늘렸을 때 시간이 줄어드는지 본다.

현재 Windows 빌드에서는 실제 소켓 서버가 열리지 않는다.

실제 발표 시연은 **Linux 또는 WSL**에서 하는 것이 안전하다.

## 4. 테스트 1: HTTP 단위 테스트

```bash

```

아무 에러 없이 끝나면 성공이다.

조금 더 쉽게 말하면:

## 5. 테스트 2: 전체 핵심 테스트 한 번에 돌리기

시간이 있으면 아래 명령도 실행한다.

```powershell
mingw32-make test
```

WSL/Linux에서는 이렇게 실행한다.

```bash
make test
```

정상이라면 아래처럼 `[PASS]`가 나온다.

```text
[PASS] bptree
[PASS] index
[PASS] parser
[PASS] schema
[PASS] executor
[PASS] http
```

여기서 발표 때 중요하게 말할 것은 전부가 아니라 아래 2개다.

- `executor`: SQL이 실제 데이터 파일과 연결되어 실행되는지 확인
- `http`: API 요청/응답 모양이 맞는지 확인

나머지 `bptree`, `index`, `parser`, `schema`는 이전 차수 DB 엔진이 깨지지 않았는지 보는 기본 안전장치다.

## 8. 테스트 5: SQL을 API로 보내기

### 8-1. INSERT 요청

```bash
curl -i -X POST http://127.0.0.1:8080/query \
  -H "Content-Type: application/json" \
  -d "{\"sql\":\"INSERT INTO users VALUES (1, 'alice', 30, 'alice@example.com');\"}"
```

성공 기준:

- HTTP 상태 코드가 `200`
- `"ok":true`
- `"rows_affected":1`

뜻:

> `alice`라는 사용자를 DB에 넣었다.

### 8-2. SELECT 요청

```bash
curl -i -X POST http://127.0.0.1:8080/query \
  -H "Content-Type: application/json" \
  -d "{\"sql\":\"SELECT * FROM users WHERE id = 1;\",\"include_profile\":true}"
```

성공 기준:

- HTTP 상태 코드가 `200`
- `"ok":true`
- `"row_count":1`
- 결과 안에 `alice`가 있음
- `"profile"`이 있으면 실행 경로도 확인 가능

뜻:

> API로 SQL을 보냈고, DBMS가 데이터를 찾아서 JSON으로 돌려줬다.

## 10. 테스트 7: Thread Pool 동시 요청 확인

수요코딩회 요구사항에는 **스레드 풀로 병렬 처리**가 있다.

그래서 여러 요청을 동시에 보내는 테스트를 한다.

서버가 켜진 상태에서 Windows PowerShell을 열고 실행한다.

프로젝트 루트에 있을 때:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\threadpool_smoke.ps1
```

`scripts` 폴더 안에 있을 때:

```powershell
powershell -ExecutionPolicy Bypass -File .\threadpool_smoke.ps1
```

이 스크립트가 하는 일:

1. `/health`로 서버가 살아있는지 확인한다.
2. 테스트용 user 데이터를 여러 개 넣는다.
3. 여러 `SELECT` 요청을 동시에 보낸다.
4. 모든 요청이 성공했는지 확인한다.

참고:

- 이 스크립트는 내부에서 `curl.exe`로 JSON을 보낸다.
- PowerShell의 `Invoke-RestMethod`를 직접 쓰면 작은따옴표가 `\u0027`처럼 바뀌어 현재 C JSON 파서가 못 읽을 수 있다.
- 그래서 동시 요청 테스트는 직접 명령을 조합하지 말고 이 스크립트로 확인하는 것이 안전하다.

성공하면 마지막에 이렇게 나온다.

```text
Thread pool smoke test passed.
```

쉽게 말하면:

> 친구 여러 명이 동시에 "내 데이터 찾아줘!"라고 말해도 서버가 처리하는지 보는 테스트다.

요청 개수를 바꾸고 싶으면 이렇게 실행한다.

프로젝트 루트에 있을 때:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\threadpool_smoke.ps1 -ConcurrentRequests 16
```

`scripts` 폴더 안에 있을 때:

```powershell
powershell -ExecutionPolicy Bypass -File .\threadpool_smoke.ps1 -ConcurrentRequests 16
```

만약 연결 실패 메시지가 나오면 스크립트 문제가 아니라 API 서버가 아직 안 켜진 것이다.
먼저 Linux/WSL에서 `./sqlp --server 8080 --threads 4`를 실행해 둔 뒤 다시 시도한다.

## 11. 테스트 8: 16/32/64/128 동시 요청 따로 확인

이 테스트는 동시 요청 수를 하나씩 따로 실행하면서 어느 지점부터 처리량이 좋아지는지 확인한다.
한 번에 `16 → 32 → 64 → 128`을 연속 실행하면 서버 상태와 클라이언트 작업 생성 비용이 누적될 수 있으므로, 발표용으로는 아래처럼 각각 따로 실행하는 것이 더 깔끔하다.

서버가 켜진 상태에서 Windows PowerShell을 열고 실행한다.

`scripts` 폴더 안에 있을 때:

```powershell
powershell -ExecutionPolicy Bypass -File .\threadpool_16.ps1
powershell -ExecutionPolicy Bypass -File .\threadpool_32.ps1
powershell -ExecutionPolicy Bypass -File .\threadpool_64.ps1
powershell -ExecutionPolicy Bypass -File .\threadpool_128.ps1
```

프로젝트 루트에 있을 때:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\threadpool_16.ps1
powershell -ExecutionPolicy Bypass -File .\scripts\threadpool_32.ps1
powershell -ExecutionPolicy Bypass -File .\scripts\threadpool_64.ps1
powershell -ExecutionPolicy Bypass -File .\scripts\threadpool_128.ps1
```

각 스크립트가 실행하는 요청 수:

| 스크립트 | 동시 요청 수 |
|---|---:|
| `threadpool_16.ps1` | 16개 |
| `threadpool_32.ps1` | 32개 |
| `threadpool_64.ps1` | 64개 |
| `threadpool_128.ps1` | 128개 |

출력 예시:

```text
========== Thread Pool Single Test: 32 concurrent request(s) ==========

requests ok fail elapsed_ms throughput_req_s avg_ms_per_request status note
-------- -- ---- ---------- ---------------- ------------------ ------ ----
32       32 0    11864      2.70             370.75             PASS   all ok
```

여기서 볼 것:

- `ok`가 요청 수와 같은지 확인한다.
- `fail`이 `0`인지 확인한다.
- `throughput_req_s`가 높을수록 같은 시간에 더 많은 요청을 성공 처리한 것이다.
- `First near-peak point`는 최고 처리량의 90% 이상이 처음 나온 지점이다.

컬럼 뜻:

| 컬럼 | 뜻 |
|---|---|
| `requests` | 동시에 보낸 `SELECT` 요청 수 |
| `ok` | 성공한 요청 수 |
| `fail` | 실패한 요청 수 |
| `elapsed_ms` | `SELECT` 동시 요청 묶음이 끝날 때까지 걸린 전체 시간 |
| `throughput_req_s` | 1초에 성공 처리한 요청 수 |
| `avg_ms_per_request` | 요청 1개당 평균 시간처럼 보기 위한 값, `elapsed_ms / requests` |
| `status` | 해당 요청 수 테스트의 통과 여부 |
| `note` | 실패했을 때 첫 번째 대표 에러 |

만약 처음 `/health` 확인에서 멈추거나 timeout이 나면, 테스트 스크립트 문제가 아니라 서버가 응답을 못 주는 상태다.
그럴 때는 서버 터미널에서 `Ctrl+C`로 끄고 다시 켠다.

```bash
./sqlp --server 8080 --threads 4
```

권장 실행 방식:

1. `threadpool_16.ps1` 실행
2. 결과 기록
3. 서버가 이상하면 재시작
4. `threadpool_32.ps1` 실행
5. 같은 방식으로 `64`, `128`도 확인

세부 row 결과까지 보고 싶으면 `-ShowRows`를 붙인다.

```powershell
powershell -ExecutionPolicy Bypass -File .\threadpool_32.ps1 -ShowRows
```

해석 방법:

```text
16개보다 32개, 64개에서 throughput_req_s가 올라가면
동시 요청이 많아질수록 worker thread를 더 잘 활용한다는 뜻이다.

어느 순간부터 throughput_req_s가 더 안 오르면
그 지점부터는 worker 수, DB lock, 파일 I/O, 테스트 오버헤드 중 하나가 병목이라는 뜻이다.
```

만약 `64`나 `128`에서 `Received HTTP/0.9 when not allowed`가 나오면, SQL 조회 실패라기보다 높은 동시 접속에서 HTTP 응답이 정상적인 `HTTP/1.1 ...` 헤더로 시작하지 못한 것이다.
이 경우는 현재 서버나 테스트 클라이언트가 그 동시 요청 수를 안정적으로 감당하지 못한 한계 지점으로 보면 된다.

중요:

> 이 테스트는 `--threads 4` 서버 하나에서 동시 요청 수를 바꿔 보는 테스트다.
> worker 1개 서버와 worker 4개 서버의 직접 속도 비교는 다음 `test_threadpool` 테스트가 더 적합하다.

## 12. 테스트 9: Worker 로그로 병렬 처리 눈으로 보기

이 테스트는 **서버 터미널에 worker 로그를 찍어서** 실제 여러 worker가 요청을 나눠 처리하는지 보는 방법이다.

먼저 서버를 다시 빌드한다.

```bash
make all
```

서버를 trace 모드로 켠다.

```bash
MINIDBMS_TRACE_WORKERS=1 ./sqlp --server 8080 --threads 4
```

조금 더 눈에 잘 보이게 하고 싶으면 요청마다 0.5초씩 일부러 늦춘다.

```bash
MINIDBMS_TRACE_WORKERS=1 MINIDBMS_WORKER_DELAY_MS=500 ./sqlp --server 8080 --threads 4
```

그 다음 Windows PowerShell에서 동시 요청 테스트를 실행한다.

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\threadpool_smoke.ps1
```

서버 터미널에 이런 로그가 보이면 된다.

```text
[worker 140123...] start request_id=select-856373651 sql="SELECT * FROM users WHERE id = 856373651;"
[worker 140124...] start request_id=select-856373652 sql="SELECT * FROM users WHERE id = 856373652;"
[worker 140125...] start request_id=select-856373653 sql="SELECT * FROM users WHERE id = 856373653;"
[worker 140126...] start request_id=select-856373654 sql="SELECT * FROM users WHERE id = 856373654;"
```

여기서 볼 것:

- `worker` 번호가 여러 개면 여러 스레드가 일한 것이다.
- `request_id=select-...`는 PowerShell 결과표의 `request_id`와 연결된다.
- 여러 `start`가 먼저 나오고 나중에 `done`이 나오면 요청들이 겹쳐서 처리된 것이다.

쉽게 말하면:

> 계산대 직원이 4명 있고, 손님 8명이 줄을 섰을 때 직원 4명이 동시에 손님을 받는 장면을 로그로 보는 테스트다.

## 13. 테스트 10: 1개 worker vs 4개 worker 시간 비교

이 테스트는 서버 내부 thread pool만 직접 확인하는 C 테스트다.
HTTP나 PowerShell 속도는 빼고, thread pool 자체가 병렬로 일하는지 본다.

Linux/WSL에서 실행한다.

```bash
make test_threadpool
./test_threadpool
```

이 테스트가 하는 일:

1. 테스트용 테이블 `tp_smoke`를 만든다.
2. `SELECT` 작업 8개를 thread pool에 넣는다.
3. worker 수를 `1`, `2`, `4`로 바꿔가며 같은 작업을 실행한다.
4. 처리 시간이 줄어드는지 확인한다.

예상 출력 모양:

```text
[1] Correctness: 8 concurrent SELECT jobs with 4 workers
  success: 8 / 8
  result: PASS

[2] Timing: same 8 jobs, different worker counts
  workers  elapsed(ms)  speedup vs 1 worker
  1        960          1.00x
  2        480          2.00x
  4        240          4.00x
  result: PASS - 4 workers finished clearly faster

[3] Completion waves with 4 workers
  first wave : 120ms 121ms 121ms 122ms
  second wave: 240ms 241ms 242ms 242ms
```

숫자가 꼭 똑같을 필요는 없다.
중요한 것은 아래 2개다.

- `success: 8 / 8`
- `4 workers`가 `1 worker`보다 확실히 빠름

왜 이게 병렬 증거인가:

```text
worker 1개면 8개 작업을 하나씩 처리한다.
worker 4개면 4개 작업을 먼저 같이 처리하고, 그 다음 4개를 처리한다.
그래서 완료 시간이 8칸짜리 한 줄이 아니라 4개씩 2묶음처럼 보인다.
```

발표에서는 이렇게 말하면 된다.

```text
외부 API 테스트는 8개 요청이 모두 성공하는지 확인했고,
내부 thread pool 테스트는 worker 수를 1개, 2개, 4개로 바꿔 같은 8개 작업의 처리 시간을 비교했습니다.
4개 worker에서 시간이 확실히 줄어 병렬 처리 흐름을 확인했습니다.
```

저희는 테스트를 네 단계로 나눴습니다.

## 15. 필요 없어서 뺀 테스트

이 문서에서는 아래 테스트 설명을 자세히 다루지 않는다.

- B+Tree 내부 split 테스트
- index manager 내부 rebuild 테스트
- parser 문법 세부 테스트
- schema 타입 세부 테스트

이유:

> 수요코딩회 발표의 핵심은 "API 서버 + 스레드 풀 + 외부 클라이언트 검증"이기 때문이다.

단, 위 테스트들은 코드 안전장치로는 여전히 좋다.
그래서 완전히 버리는 것이 아니라 `mingw32-make test` 안에서 같이 돌릴 수 있게만 둔다.
