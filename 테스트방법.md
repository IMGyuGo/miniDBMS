# 테스트 방법

## 1. 현재 검증 상태

현재 Windows 기준으로 아래 범위는 직접 빌드하고 실행해서 확인했다.

- `test_bptree`
- `test_index`
- `test_parser`
- `test_schema`
- `test_executor`
- `test_http`
- `sqlp.exe` CLI 빌드
- `sqlp.exe` 스모크 테스트

즉, 현재는 `CLI SQL 엔진 + 인덱스 경로` 까지는 동작 확인이 끝난 상태다.

다만 아래 항목은 아직 별도 테스트 파일이 없거나 Windows에서 런타임 검증이 끝나지 않았다.

- `tests/test_threadpool.c`
- `tests/test_api.c`
- `tests/test_concurrency.c`
- 실제 HTTP 서버 런타임 end-to-end 검증

또한 현재 Windows 빌드에서는 `--server` 모드가 스텁 처리되어 있다. 서버 런타임 검증은 Linux/WSL 환경에서 진행하는 것이 안전하다.

## 2. 가장 빠른 확인

프로젝트 루트에서 아래 명령을 실행한다.

```powershell
mingw32-make test
```

이 명령은 아래 테스트를 한 번에 빌드하고 실행한다.

- B+Tree 단위 테스트
- index manager 단위 테스트
- parser 단위 테스트
- schema 단위 테스트
- executor 통합 테스트
- HTTP 메시지 테스트

정상이라면 각 테스트 뒤에 `[PASS]` 가 출력되고 중간에 멈추지 않는다.

참고:

- schema/parser 테스트는 의도적으로 실패 케이스도 검증하므로 `stderr` 에 에러 메시지가 보일 수 있다.
- 중요한 기준은 최종 종료 코드와 `[PASS]` 여부다.

## 3. 전체 빌드 확인

CLI 실행 파일까지 확인하려면 아래 명령을 실행한다.

```powershell
mingw32-make all
```

정상 빌드되면 프로젝트 루트에 `sqlp.exe` 가 생성된다.

## 4. CLI 스모크 테스트

### 4-1. 테스트 SQL 파일 만들기

```powershell
@"
INSERT INTO users VALUES (1, 'alice', 30, 'alice@example.com');
INSERT INTO users VALUES (2, 'bob', 25, 'bob@example.com');
SELECT * FROM users WHERE id = 2;
SELECT * FROM users WHERE age BETWEEN 25 AND 30;
"@ | Set-Content -Encoding ascii .\smoke_test.sql
```

### 4-2. 실행

```powershell
.\sqlp.exe .\smoke_test.sql
```

### 4-3. 기대 결과

아래 흐름이 보이면 정상이다.

- `INSERT` 2건 성공
- `WHERE id = 2` 에서 `bob` 행 1건 조회
- `WHERE age BETWEEN 25 AND 30` 에서 `alice`, `bob` 2건 조회
- `stderr` 에 `[SELECT][index:id:eq ...]`, `[SELECT][index:age:range ...]` 같은 로그가 보일 수 있음

### 4-4. 정리

스모크 테스트 후 임시 파일을 지우려면 아래 명령을 실행한다.

```powershell
Remove-Item .\smoke_test.sql -ErrorAction SilentlyContinue
Remove-Item .\data\users.dat -ErrorAction SilentlyContinue
```

## 5. API 서버 실제 확인 방법

### 5-1. 현재 Windows 빌드에서 확인 가능한 것

현재 Windows 빌드의 `--server` 모드는 실제 소켓 서버를 열지 않고 스텁으로 막아두었다.

```powershell
.\sqlp.exe --server 8080
```

현재 상태에서는 아래 메시지가 나오면 정상이다.

```text
[server] --server mode is not supported in this Windows build yet.
```

즉, Windows에서는 "서버 모드 진입 가드가 걸려 있는지" 까지만 확인할 수 있다.

### 5-2. 실제 API 서버를 띄워 확인하려면

실제 HTTP 서버 검증은 Linux 또는 WSL 환경에서 진행한다.

깨끗한 상태에서 확인하려면 서버 실행 전에 테스트 데이터 파일을 지운다.

```bash
rm -f ./data/users.dat
```

터미널 1에서 프로젝트 루트로 이동한 뒤 서버를 실행한다.

```bash
make all
./sqlp --server 8080 --threads 4
```

정상 기동되면 아래와 비슷한 로그가 `stderr` 에 보인다.

```text
[server] miniDBMS HTTP server listening on port 8080 with 4 worker thread(s)
[server] POST /query  — SQL 실행
[server] GET  /health — 서버 상태
```

### 5-3. Health check

터미널 2에서 아래 요청을 보낸다.

```bash
curl -i http://127.0.0.1:8080/health
```

기대 결과:

- HTTP status `200`
- JSON body 에 `"ok":true`
- JSON body 에 `"code":"OK"`
- JSON body 에 `"message":"healthy"`

예시:

```json
{"ok":true,"http_status":200,"code":"OK","message":"healthy"}
```

### 5-4. Query API 기본 확인

먼저 `INSERT` 를 한 번 보낸다.

```bash
curl -i -X POST http://127.0.0.1:8080/query \
  -H "Content-Type: application/json" \
  -d "{\"sql\":\"INSERT INTO users VALUES (1, 'alice', 30, 'alice@example.com');\"}"
```

기대 결과:

- HTTP status `200`
- `"ok":true`
- `"rows_affected":1`
- `"has_payload":false`

그 다음 `SELECT` 를 보낸다.

```bash
curl -i -X POST http://127.0.0.1:8080/query \
  -H "Content-Type: application/json" \
  -d "{\"sql\":\"SELECT * FROM users WHERE id = 1;\",\"include_profile\":true}"
```

기대 결과:

- HTTP status `200`
- `"ok":true`
- `"has_payload":true`
- `"row_count":1`
- `"rows"` 배열 안에 `id`, `name`, `age`, `email` 이 포함됨
- `include_profile` 을 켠 경우 `"profile"` 객체가 포함됨

응답 예시:

```json
{
  "ok": true,
  "http_status": 200,
  "code": "OK",
  "row_count": 1,
  "rows_affected": 0,
  "has_payload": true,
  "error": "",
  "rows": [
    {
      "id": "1",
      "name": "alice",
      "age": "30",
      "email": "alice@example.com"
    }
  ],
  "profile": {
    "elapsed_ms": 0.000,
    "tree_io": 1,
    "row_count": 1,
    "access_path": "index:id:eq"
  }
}
```

### 5-5. 인덱스 경로와 선형 경로 비교 확인

`force_linear` 옵션을 켜면 인덱스 대신 선형 경로를 강제할 수 있다.

```bash
curl -i -X POST http://127.0.0.1:8080/query \
  -H "Content-Type: application/json" \
  -d "{\"sql\":\"SELECT * FROM users WHERE age BETWEEN 25 AND 30;\",\"force_linear\":true,\"include_profile\":true}"
```

이 경우 응답의 `profile.access_path` 가 인덱스 경로가 아닌 선형 경로로 나오는지 확인한다.

### 5-6. 실패 케이스 확인

잘못된 경로나 메서드도 같이 확인하면 좋다.

지원하지 않는 route:

```bash
curl -i -X POST http://127.0.0.1:8080/unknown -d "{}"
```

기대 결과:

- HTTP status `404`
- `"code":"UNSUPPORTED_ROUTE"`

잘못된 method:

```bash
curl -i http://127.0.0.1:8080/query
```

기대 결과:

- HTTP status `405`
- `"code":"UNSUPPORTED_METHOD"`

빈 body:

```bash
curl -i -X POST http://127.0.0.1:8080/query \
  -H "Content-Type: application/json" \
  -d ""
```

기대 결과:

- HTTP status `400`
- `"code":"BAD_REQUEST"`

### 5-7. 종료

서버를 띄운 터미널에서 `Ctrl+C` 로 종료한다.

## 6. 테스트가 확인하는 범위

### `test_bptree`

- 삽입/검색
- 노드 split 이후 검색 유지
- duplicate key 처리
- range 검색

### `test_index`

- `index_init()` 초기화와 멱등성
- `.dat` 파일 rebuild
- malformed row skip
- `id`, `age` 인덱스 검색
- I/O stats 경로
- 최대 테이블 수 제한

### `test_parser`

- 토큰 기반 SQL 파싱
- `SELECT`, `INSERT`, `WHERE`, `BETWEEN` 구문 처리
- 잘못된 구문 reject

### `test_schema`

- schema 로딩
- 컬럼 타입 검증
- `BETWEEN` 이 INT 컬럼에만 허용되는지 확인

### `test_executor`

- `INSERT` 후 파일 append
- offset 기반 조회
- `WHERE id = ...`
- `WHERE age BETWEEN ...`
- `db_service_execute_sql()` 통합 경로

### `test_http`

- HTTP request/response 메시지 파싱
- 상태 코드/헤더/바디 포맷 검증

## 7. 실제 실행 흐름

현재 CLI 경로는 아래 순서로 동작한다.

```text
SQL 파일
  -> src/input/*
  -> src/parser/*
  -> src/schema/*
  -> src/executor/*
  -> src/index/*
  -> src/bptree/*
  -> data/{table}.dat
```

좀 더 자세히 보면 다음과 같다.

1. `src/main.c`
   - SQL 파일을 읽고 statement 단위로 분리한다.

2. `src/input/lexer.c`
   - SQL 문자열을 token 목록으로 만든다.

3. `src/parser/parser.c`
   - token 목록을 AST(`SELECT`, `INSERT`) 로 바꾼다.

4. `src/schema/schema.c`
   - `schema/{table}.schema` 를 읽고 AST 가 스키마와 맞는지 검증한다.

5. `src/executor/executor.c`
   - `INSERT` 이면 `data/{table}.dat` 끝에 row 를 append 한다.
   - append 직전 file offset 을 잡아서 인덱스에도 같이 반영한다.
   - `SELECT` 이면 조건에 따라 index 경로 또는 linear scan 경로를 선택한다.

6. `src/index/index_manager.c`
   - `id`, `age` 값을 file offset 으로 연결한다.
   - 필요하면 `.dat` 파일을 다시 읽어서 index rebuild 를 수행한다.

7. `src/bptree/bptree.c`
   - 실제 key -> offset 검색 구조를 담당한다.

8. 다시 `src/executor/executor.c`
   - index 가 준 offset 으로 `.dat` 파일에서 row 를 읽어 `ResultSet` 을 만든다.

## 8. 서버 경로는 어떻게 다른가

서버 모드에서는 아래 경로가 추가된다.

```text
HTTP request
  -> src/http/*
  -> src/service/*
  -> src/executor/*
  -> src/index/*
  -> src/bptree/*
```

- `src/http/*` 는 HTTP 메시지 파싱/응답 포맷을 맡는다.
- `src/service/db_service.c` 는 SQL 엔진 호출 경계를 고정한다.
- `src/server/*`, `src/threadpool/*` 는 서버 런타임과 worker 실행을 맡는다.

현재 Windows 빌드에서는 서버 런타임을 실제로 열지 않으므로, 이 경로의 최종 검증은 Linux/WSL 에서 별도로 진행해야 한다.

## 9. 문제를 볼 때 체크할 포인트

- `mingw32-make test` 가 중간에 멈추면 먼저 어느 테스트에서 멈췄는지 확인한다.
- `test_executor` 가 깨지면 `data/*.dat` offset 과 `index_insert_*()` 경로를 먼저 본다.
- `test_index` 가 깨지면 rebuild 입력 파일 형식과 malformed row 처리부터 본다.
- CLI 스모크에서 `SELECT` 결과가 이상하면 `schema/users.schema`, `data/users.dat`, index rebuild 여부를 같이 확인한다.

## 10. 발표용 최소 산출물

수요코딩회 기준으로 실제 시연에 바로 쓸 수 있는 파일은 아래 두 개다.

- `postman/miniDBMS.postman_collection.json`
- `scripts/threadpool_smoke.ps1`

### Postman 확인

1. 서버를 먼저 켠다.
2. Postman 에서 `postman/miniDBMS.postman_collection.json` 를 import 한다.
3. `base_url` 을 서버 주소로 맞춘다.
4. 아래 순서로 실행한다.

- `Health`
- `Insert User`
- `Select By Id`
- `Force Linear Range`
- `Invalid Route`

### Thread pool 스모크 확인

서버가 켜진 상태에서 Windows PowerShell 에서 아래를 실행한다.

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\threadpool_smoke.ps1
```

이 스크립트는 다음 순서로 동작한다.

- `/health` 확인
- 테스트 row seed insert
- 여러 `SELECT` 요청 동시 전송
- 각 응답의 성공 여부와 `access_path` 출력

모든 요청이 성공하면 마지막에 `Thread pool smoke test passed.` 가 출력된다.

## 11. 빌드 산출물 정리

빌드 후 생기는 `.exe` 파일은 두 종류로 보면 된다.

- `sqlp.exe`
  - 실제 CLI 실행 파일
  - 서버/CLI 시연에 필요할 수 있음
- `test_*.exe`
  - 테스트 실행용 임시 산출물
  - 다시 빌드하면 언제든 재생성됨
  - 지워도 됨

한 번에 지우려면 아래 명령을 쓴다.

```powershell
mingw32-make clean
```
