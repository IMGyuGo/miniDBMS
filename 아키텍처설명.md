# 아키텍처 설명

## 1. 문서 목적

이 문서는 `miniDBMS` 저장소의 현재 폴더 트리를 기준으로,

- 어떤 디렉터리가 핵심 소스인지
- 어떤 파일이 실행 흐름을 이루는지
- 어떤 폴더가 테스트/문서/보조 도구 역할을 하는지
- 현재 구조에서 API 서버로 확장하려면 어디를 기준으로 보면 되는지

를 빠르게 이해할 수 있도록 정리한 안내서다.

---

## 2. 프로젝트 한 줄 요약

이 프로젝트는 **C 언어로 만든 파일 기반 미니 DBMS**다.

- SQL 파일을 읽는다.
- `lexer -> parser -> schema -> executor` 흐름으로 처리한다.
- 데이터는 `data/{table}.dat` 파일에 저장한다.
- 조회 최적화를 위해 `id`, `age` 기준의 **B+ Tree 인덱스**를 사용한다.
- 현재 기본 진입점은 CLI 프로그램(`sqlp`, `sqlp_sim`)이다.

즉, 지금 구조는 **"DB 엔진은 이미 있고, 바깥 입출력 레이어를 CLI로 붙여 둔 상태"** 로 볼 수 있다.

## 5. 실행 흐름 기준 아키텍처

이 프로젝트의 핵심 흐름은 폴더 단위로 보면 아래와 같다.

```text
samples/*.sql or 사용자 SQL 파일
    ↓
src/input/input.c         : 파일 전체 읽기
    ↓
src/input/lexer.c         : SQL 문자열을 토큰으로 분해
    ↓
src/parser/parser.c       : 토큰을 AST로 파싱
    ↓
src/schema/schema.c       : schema/{table}.schema 기준으로 유효성 검사
    ↓
src/executor/executor.c   : INSERT / SELECT 실행
    ↓
src/index/index_manager.c : 테이블별 인덱스 관리
    ↓
src/bptree/bptree.c       : B+ Tree 탐색/삽입/범위 조회
    ↓
data/{table}.dat          : 실제 row 저장 파일
```

이 흐름을 잡고 보면, 각 폴더의 역할이 명확해진다.

## 6. 폴더별 상세 설명

### 6-1. `include/` — 모듈 사이 계약

이 폴더는 각 구현체가 공유하는 **헤더 계약**을 모아 둔 곳이다.

### `include/interface.h`

가장 중요한 공통 인터페이스 파일이다.

- 토큰 타입(`TokenType`)
- 토큰 목록(`TokenList`)
- AST 구조(`ASTNode`, `SelectStmt`, `InsertStmt`)
- WHERE 절 구조(`WhereClause`)
- 스키마 구조(`TableSchema`)
- 실행 결과 구조(`ResultSet`, `Row`)

즉, `input/parser/schema/executor`가 서로 데이터를 주고받는 형식을 정의한다.

### `include/bptree.h`

B+ Tree 모듈의 공개 API다.

- `bptree_create`
- `bptree_insert`
- `bptree_search`
- `bptree_range`
- `bptree_height`

인덱스 매니저는 이 API를 통해 트리를 사용한다.

### `include/index_manager.h`

테이블별 인덱스를 관리하는 API다.

- `index_init`
- `index_insert_id`
- `index_search_id`
- `index_range_id`
- `index_insert_age`
- `index_range_age`

실행기(`executor`)는 직접 B+ Tree를 건드리지 않고, 이 계층을 통해 접근한다.

즉, `include/`는 **"모듈 경계선"** 역할을 한다.

---

### 6-2. `src/` — 실제 엔진 구현

`src/`는 현재 프로젝트의 핵심 구현이 모여 있는 곳이다.

### `src/main.c`

현재 프로젝트의 **CLI 진입점**이다.

- SQL 파일 경로를 입력받는다.
- 전체 SQL 파일을 읽는다.
- statement 단위로 나눈다.
- 파싱, 검증, 실행을 순서대로 호출한다.
- `--force-linear`, `--compare` 같은 실행 모드를 처리한다.

즉, 현재는 서버가 아니라 **명령행 프로그램 중심 구조**다.

### `src/input/`

입력 전처리 계층이다.

#### `src/input/input.c`

- SQL 파일을 바이너리 모드(`rb`)로 읽어 문자열로 반환한다.

#### `src/input/lexer.c`

- SQL 문자열을 토큰 목록으로 변환한다.
- `SELECT`, `INSERT`, `WHERE`, `BETWEEN`, `AND` 같은 키워드를 인식한다.

이 폴더는 **"문자열 입력 -> 토큰"** 단계다.

### `src/parser/`

#### `src/parser/parser.c`

- 토큰 목록을 AST로 바꾼다.
- `SELECT`
- `INSERT`
- `WHERE col = value`
- `WHERE col BETWEEN a AND b`

같은 SQL 구조를 해석한다.

이 폴더는 **"토큰 -> 의미 있는 문장 구조(AST)"** 단계다.

### `src/schema/`

#### `src/schema/schema.c`

- `schema/{table}.schema` 파일을 읽는다.
- 테이블 컬럼 정의를 메모리 구조로 로딩한다.
- SQL 문이 스키마와 맞는지 검사한다.

예를 들어:

- 존재하지 않는 컬럼 사용
- 타입이 맞지 않는 값 사용
- `BETWEEN`을 정수형 컬럼이 아닌 곳에 쓰는 경우

같은 문제를 잡아낸다.

이 폴더는 **"쿼리가 테이블 정의와 맞는지 검사하는 계층"** 이다.

### `src/executor/`

#### `src/executor/executor.c`

실제 쿼리를 수행하는 핵심 계층이다.

- `INSERT`면 `data/{table}.dat`에 row를 append
- append 시작 offset을 계산
- `id`, `age` 인덱스 갱신
- `SELECT`면 조건을 보고 인덱스 사용 여부 결정
- 인덱스를 쓰지 못하면 선형 탐색
- 결과를 `ResultSet`으로 만들어 반환

즉, 이 파일은 **"쿼리 의미(AST)와 실제 데이터 파일/인덱스 사이를 연결하는 브리지"** 다.

#### `src/executor/executor_internal.h`

외부 공개 API는 아니고, 실행기 내부에서 쓰는 보조 구조를 담는다.

- 선택 경로
- 수행 시간
- 트리 I/O 수
- 결과 row 수

같은 측정 정보를 담는 `SelectExecInfo`가 정의되어 있다.

### `src/index/`

#### `src/index/index_manager.c`

테이블 단위로 인덱스를 초기화하고 관리하는 계층이다.

- 테이블별 `id` 인덱스
- 테이블별 `age` 인덱스
- 검색/범위 조회 API
- 트리 높이, I/O 통계 조회

즉, `executor` 입장에서는 B+ Tree 구현 세부사항을 몰라도 되도록 중간 레이어를 제공한다.

### `src/bptree/`

#### `src/bptree/bptree.c`

가장 하단의 인덱스 자료구조 구현체다.

- B+ Tree 생성
- 삽입
- 단건 탐색
- 범위 탐색
- 높이 조회
- I/O 시뮬레이션

현재 프로젝트의 성능 비교 핵심은 이 폴더에 있다.

---

### 6-3. `schema/` — 테이블 정의

### `schema/users.schema`

현재 엔진이 사용하는 실제 테이블 정의 파일이다.

현재 내용은 아래와 같다.

```text
table=users
columns=4
col0=id,INT,0
col1=name,VARCHAR,64
col2=age,INT,0
col3=email,VARCHAR,128
```

즉, 이 파일은 다음 정보를 제공한다.

- 테이블 이름: `users`
- 총 컬럼 수: 4
- 인덱스 후보 정수 컬럼: `id`, `age`
- 문자열 컬럼: `name`, `email`

`schema_load()`는 이 파일을 읽어 메모리 구조로 만들고,
`schema_validate()`는 이 정의를 기준으로 쿼리의 컬럼/타입 적합성을 검사한다.

즉, `schema/`는 **코드가 아니라 메타데이터** 폴더다.

## 9. API 서버 과제로 확장할 때 어디를 건드리게 되는가

현재 폴더 트리 기준으로 보면, API 서버는 **엔진 바깥 레이어를 추가하는 방식**이 가장 자연스럽다.

추천 관점은 아래와 같다.

### 그대로 재사용할 가능성이 큰 영역

- `include/`
- `src/input/`
- `src/parser/`
- `src/schema/`
- `src/executor/`
- `src/index/`
- `src/bptree/`

즉, DB 엔진 본체는 최대한 유지하는 쪽이 좋다.

### 새로 생기기 쉬운 영역

- `src/server/`
- `src/api/`
- `src/threadpool/`
- `tests/test_api*.c`
- `samples/*.http` 또는 API 테스트용 샘플

### 가장 큰 구조 변화 지점

- 현재 `src/main.c`는 CLI 진입점이다.
- API 서버 과제에서는 여기에 더해 **소켓 서버 진입점** 또는 **별도 서버 main** 이 필요할 가능성이 높다.

즉, 지금 프로젝트는 이미 DB 엔진으로는 구조가 잡혀 있고,
앞으로는 그 위에 **네트워크 요청 처리 계층**을 얹는 방향으로 보는 것이 맞다.
