## 아키텍처 (약 40초)

전체 구조는 이렇습니다.

클라이언트가 HTTP로 SQL을 보내면, HTTP Server가 요청을 받아서 잡 큐에 넣습니다. 워커 스레드 4개가 큐에서 잡을 꺼내 DB 엔진을 호출하고, 결과를 JSON으로 돌려줍니다.

서버와 엔진은 `db_service_execute_sql()` 하나로만 연결됩니다. HTTP 계층이 엔진 내부를 직접 건드리지 않고, 이 함수를 통해서만 접근하도록 계층을 분리했습니다.

## 스레드 풀 (약 1분 10초)

핵심 구현인 스레드 풀 설명드리겠습니다.

요청마다 스레드를 새로 만들면 OS 자원 비용이 생기고, 요청이 폭발적으로 늘면 스레드도 같이 폭발합니다. 그래서 서버 시작 시 워커 스레드 4개를 미리 만들어두고 재사용하는 구조를 선택했습니다.

요청 처리 흐름을 보면, HTTP Server는 요청이 오면 mutex로 잡 큐를 잠그고 SQL을 적재한 뒤 `cond_signal`로 워커 하나를 깨웁니다. 그리고 바로 다음 요청을 받으러 돌아갑니다.

깨어난 워커는 큐에서 잡을 꺼내고, SQL 종류를 판별합니다. 여기서 rwlock이 적용됩니다.

SELECT면 rdlock, 읽기 잠금을 겁니다. 읽기끼리는 서로 방해가 없어서 워커 여러 개가 동시에 실행할 수 있습니다.

INSERT면 wrlock, 쓰기 잠금입니다. 이 워커 혼자만 엔진에 접근하고 나머지는 대기합니다. 동시에 파일을 쓰면 데이터가 깨질 수 있기 때문입니다.

mutex 하나로만 처리했다면 SELECT끼리도 줄을 서야 해서 불필요한 대기가 생깁니다. rwlock으로 읽기는 동시에, 쓰기만 단독으로 처리해 성능을 높였습니다.

워커는 잡이 없을 때 `pthread_cond_wait`으로 실제로 잠듭니다. 루프를 돌며 CPU를 태우는 스핀락이 아닙니다.

## 성능 수치 (약 30초)

실제로 병렬성이 얼마나 나오는지 수치로 확인했습니다.

같은 64개 요청을 워커 수만 바꿔서 처리했을 때, 워커 1개에서 480ms 걸리던 작업이 워커 4개에서 125ms로 줄었습니다. 약 3.84배 speedup입니다.

B+Tree와 선형 탐색도 비교했는데, `WHERE id = 5001` 조건은 B+Tree로 0.136ms, 노드 2번만 방문해서 끝났습니다. 반면 `WHERE age = 25`는 인덱스를 타지 않고 파일 전체를 스캔해서 0.636ms가 나왔습니다.

## 테스트 및 엣지 케이스 (약 30초)

테스트는 스레드 풀, Executor, B+Tree 세 영역으로 나눠서 작성했습니다.

특히 동일 쿼리를 인덱스 경로와 linear 경로 두 가지로 실행해서 결과가 일치하는지 검증했고, B+Tree는 노드 분할 이후에도 기존 키가 정상 조회되는지 확인했습니다.

엣지 케이스로는 잡 큐 포화, 서버 종료 시 잠든 워커 정리, 잘못된 SQL 입력 시 서버 크래시 없이 BAD_REQUEST 반환하는 것까지 처리했습니다.
