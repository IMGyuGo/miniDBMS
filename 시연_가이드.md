# 시연 가이드

---

## 병렬 처리란

여러 작업을 동시에 처리하는 것. 반대는 순차 처리로, 하나가 끝나야 다음을 시작.

```
순차 처리 (워커 1개)
요청 A ───────► 완료
요청 B              ───────► 완료
요청 C                           ───────► 완료
시간 →  1초         2초         3초

병렬 처리 (워커 4개)
요청 A ───────► 완료
요청 B ───────► 완료
요청 C ───────► 완료
요청 D ───────► 완료
시간 →  1초
```

## 시연 순서

### 1. 서버 실행

```bash
MINIDBMS_TRACE_WORKERS=1 ./sqlp --server 8080 --threads 4
```

`MINIDBMS_TRACE_WORKERS=1` 을 붙이면 각 워커의 start/done 로그가 출력됨.

---

### 2. 헬스체크 — 서버 살아있는지 확인

```bash
curl http://localhost:8080/health
```

```json
{ "ok": true, "http_status": 200, "code": "OK", "message": "healthy" }
```

---

### 3. INSERT — 데이터 넣기

```bash
curl -X POST http://localhost:8080/query \
  -H "Content-Type: application/json" \
  -d '{"sql": "INSERT INTO users VALUES (9999, '\''demo-user'\'', 25, '\''demo@example.com'\'');"}'
```

```json
{ "ok": true, "rows_affected": 1, "has_payload": false }
```

---

### 4. SELECT — 인덱스 경로 확인

```bash
curl -X POST http://localhost:8080/query \
  -H "Content-Type: application/json" \
  -d '{"sql": "SELECT * FROM users WHERE id = 9999;"}'
```

```json
{
  "ok": true,
  "rows": [{ "id": "9999", "name": "demo-user", "age": "25", "email": "demo@example.com" }],
  "profile": {
    "elapsed_ms": 0.136,
    "tree_io": 2,
    "access_path": "index:id:eq"
  }
}
```

`access_path: index:id:eq` → B+Tree로 찾았음. `tree_io: 2` → 노드 2번 방문.

---

### 5. SELECT — linear 경로 비교

```bash
curl -X POST http://localhost:8080/query \
  -H "Content-Type: application/json" \
  -d '{"sql": "SELECT * FROM users WHERE age = 25;"}'
```

```json
{
  "profile": {
    "elapsed_ms": 0.636,
    "tree_io": 0,
    "access_path": "linear"
  }
}
```

`access_path: linear` → 파일 전체 스캔. `tree_io: 0` → B+Tree 미사용.

---

### 6. 병렬 처리 증거 — 로그로 확인

Postman 또는 스크립트로 여러 요청을 동시에 보내면 서버 로그에서 아래처럼 찍힘.

```
[worker 1] start select-105   ← 워커 1 시작
[worker 2] start select-111   ← 워커 2 동시 시작
[worker 3] start select-106   ← 워커 3 동시 시작
[worker 4] start select-109   ← 워커 4 동시 시작
[worker 3] done  select-105
[worker 1] done  select-106
[worker 4] done  select-111
[worker 2] done  select-109
```

**포인트 3가지**

- worker ID가 전부 다름 → 서로 다른 스레드가 각자 처리 중
- done 없이 start가 4개 연속 → 4개 요청이 동시에 실행 중
- done 순서가 제출 순서와 다름 → 동시에 달리는 중이라 먼저 제출됐다고 먼저 끝나지 않음

---

### 7. 병렬 처리 수치 증명 — test_threadpool

```bash
./test_threadpool 64
```

```
workers  elapsed(ms)  speedup
-------  -----------  -------
1                480   1.00x
2                245   1.96x
4                125   3.84x
```

같은 64개 요청을 worker 수만 바꿔서 처리. 수치로 병렬성 효과를 직접 확인.

## 설명 포인트 요약

| 시연 항목 | 설명할 내용 |
|-----------|-------------|
| 헬스체크 | 서버가 떠 있고 HTTP 응답을 주는 것 확인 |
| INSERT | row가 파일에 기록되고 인덱스에도 등록됨 |
| SELECT (index) | B+Tree로 찾아서 빠름. tree_io = 트리 높이 |
| SELECT (linear) | 인덱스 조건에 안 맞으면 전체 스캔. tree_io = 0 |
| 로그 | worker ID 4개가 동시에 start → 병렬 처리 증거 |
| test_threadpool | worker 수 늘릴수록 elapsed 줄어드는 수치 증명 |
