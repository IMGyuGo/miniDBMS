# 스레드 풀 (Thread Pool) 개념 정리

---

## 스레드 풀이란

요청마다 스레드를 새로 만들고 없애는 대신, 미리 고정 개수의 스레드를 만들어두고 재사용하는 구조입니다.

**스레드를 매번 생성하면 생기는 문제**

- 스레드 생성/파괴 자체에 시간이 걸립니다 (OS 자원 할당 비용)
- 요청이 폭발적으로 늘어나면 스레드도 같이 폭발해서 메모리와 CPU가 고갈됩니다

**스레드 풀이 해결하는 것**

- 스레드를 미리 만들어두므로 요청이 들어와도 생성 비용이 없습니다
- 워커 수로 동시 처리 한도를 고정해 서버를 안정적으로 유지합니다
- 잡 큐에 쌓아두고 순서대로 처리하므로 요청이 몰려도 튕기지 않습니다

---

## 구성 요소

| 구성 요소 | 역할 |
|-----------|------|
| 워커 스레드 | 잡 큐에서 작업을 꺼내 실행하는 스레드. 서버 시작 시 고정 개수(4개) 생성되고 서버가 끝날 때까지 유지 |
| 잡 큐 | 들어온 요청을 순서대로 쌓아두는 링버퍼. 워커가 앞에서 꺼내고 서버가 뒤에서 넣음 |
| mutex | 큐에 여러 스레드가 동시에 접근하지 못하도록 잠그는 열쇠 |
| cond_not_empty | 큐가 비었을 때 워커를 재우고, 잡이 들어오면 깨우는 신호 |
| cond_not_full | 큐가 꽉 찼을 때 제출자를 재우는 신호 (큐에 빈 자리가 생기면 깨움) |
| stop 플래그 | 서버 종료 시 워커에게 "이제 그만"을 알리는 신호 |

## 잡 큐 — 링버퍼 구조

```
index:   0    1    2    3    4    5    6    7
       [ A ][ B ][ C ][   ][   ][   ][   ][   ]
              ↑                   ↑
            head                tail
          (다음 꺼낼 곳)       (다음 넣을 곳)
```

- `tail`에 새 잡을 넣고, `head`에서 잡을 꺼냅니다
- 배열 끝에 도달하면 인덱스가 0으로 돌아옵니다 (`% QUEUE_MAX`)
- `queue_count`로 현재 잡 개수를 추적합니다

```c
// 잡 넣기
tp->queue[tp->queue_tail] = *job;
tp->queue_tail = (tp->queue_tail + 1) % THREADPOOL_QUEUE_MAX;
tp->queue_count++;

// 잡 꺼내기
ThreadpoolJob job = tp->queue[tp->queue_head];
tp->queue_head = (tp->queue_head + 1) % THREADPOOL_QUEUE_MAX;
tp->queue_count--;
```

## 요청 처리 흐름

```
클라이언트 요청 수신
        │
        ▼
  HTTP Server
  mutex lock
  큐에 잡 적재 (queue_tail에 넣기)
  cond_signal  ──────────────────────────► 잠든 워커 1개 깨움
  mutex unlock
        │
        │                잠들어 있던 워커
        │                        │
        │                cond_wait에서 깨어남
        │                mutex lock
        │                큐에서 잡 꺼냄 (queue_head에서 꺼내기)
        │                queue_count--
        │                cond_signal (큐 빈 자리 생겼다고 알림)
        │                mutex unlock
        │                        │
        │               SQL 종류 판별
        │                ┌────────┴────────┐
        │             SELECT            INSERT
        │            rdlock            wrlock
        │         (동시 읽기)         (단독 쓰기)
        │                └────────┬────────┘
        │               db_service_execute_sql()
        │                        │
        │                   lock unlock
        │                        │
        │                   응답 전송
        │                        │
        ▼                cond_wait (다시 대기)
```

## 워커 스레드 생명 주기

```c
// 워커 루프 핵심 구조 (threadpool.c)
for (;;) {
    pthread_mutex_lock(&tp->mutex);

    // 잡이 없으면 실제로 잠듦 (CPU 사용 0)
    while (tp->queue_count == 0 && !tp->stop)
        pthread_cond_wait(&tp->cond_not_empty, &tp->mutex);

    // stop이고 큐도 비었으면 종료
    if (tp->stop && tp->queue_count == 0) {
        pthread_mutex_unlock(&tp->mutex);
        break;
    }

    // 잡 꺼내기
    ThreadpoolJob job = tp->queue[tp->queue_head];
    tp->queue_head = (tp->queue_head + 1) % THREADPOOL_QUEUE_MAX;
    tp->queue_count--;

    pthread_cond_signal(&tp->cond_not_full);
    pthread_mutex_unlock(&tp->mutex);

    // 잡 처리 (엔진 호출)
    ...
}
```

`pthread_cond_wait`은 내부적으로 mutex를 atomic하게 unlock하고 잠들기 때문에, 잡 제출 측과 레이스 컨디션 없이 안전하게 동작합니다.

## 동시성 보호 — mutex vs rwlock

스레드 풀에는 두 종류의 잠금이 사용됩니다.

### mutex — 잡 큐 보호

잡 큐는 여러 스레드가 동시에 읽고 쓰면 데이터가 깨질 수 있습니다. mutex로 큐 접근을 한 번에 하나씩만 허용합니다.

```
큐에 잡 넣기:   lock → 넣기 → signal → unlock
큐에서 잡 꺼내기: lock → 꺼내기 → signal → unlock
```

### rwlock — DB 엔진 보호

SQL 실행 시 여러 워커가 동시에 파일을 읽거나 쓸 수 있어서 데이터 일관성이 깨질 수 있습니다. rwlock은 읽기와 쓰기를 분리해 불필요한 대기를 줄입니다.

```
SELECT  → rdlock : 여러 워커가 동시에 읽기 가능
INSERT  → wrlock : 혼자만 접근, 나머지는 대기
```

```c
int is_select = (strncasecmp(job.sql, "SELECT", 6) == 0);
if (is_select)
    pthread_rwlock_rdlock(&g_engine_rwlock);
else
    pthread_rwlock_wrlock(&g_engine_rwlock);

db_service_execute_sql(job.sql, &opts, &result);

pthread_rwlock_unlock(&g_engine_rwlock);
```

**mutex만 쓰면 안 되나?** — mutex는 읽기도 단독으로 처리합니다. SELECT가 많은 서버에서는 읽기끼리도 줄을 서야 해서 불필요한 대기가 생깁니다. rwlock을 쓰면 SELECT 요청들이 동시에 처리됩니다.

## 안전한 종료 흐름

```c
// threadpool_destroy() 핵심 흐름
pthread_mutex_lock(&tp->mutex);
tp->stop = 1;
pthread_cond_broadcast(&tp->cond_not_empty);  // 잠든 워커 전부 깨움
pthread_mutex_unlock(&tp->mutex);

// 모든 워커가 종료될 때까지 기다림
for (int i = 0; i < tp->num_threads; i++)
    pthread_join(tp->threads[i], NULL);

// 큐에 남은 잡의 fd 닫기 (리소스 정리)
while (tp->queue_count > 0) {
    if (j->conn_fd >= 0) close(j->conn_fd);
    ...
}
```

`cond_signal`이 아니라 `cond_broadcast`를 쓰는 이유는, 잠든 워커가 여러 개일 수 있어서 전부 깨워야 stop 플래그를 확인하고 종료할 수 있기 때문입니다.

## 엣지 케이스

| 상황 | 처리 방식 |
|------|-----------|
| 잡 큐가 꽉 참 | `threadpool_submit()` 0 반환, 해당 잡 버림 |
| 워커 생성 실패 | `stop = 1`로 이미 만들어진 워커 정리 후 NULL 반환 |
| 종료 시 큐에 잡 남아있음 | 워커 join 후 남은 잡의 fd를 순서대로 닫음 |
| 잡이 없는데 stop 신호 | `cond_broadcast`로 깨운 후 워커가 stop 조건 확인하고 종료 |
| NULL 풀 포인터 전달 | `threadpool_submit()`, `threadpool_destroy()` 모두 즉시 반환 |

---

## 정리

```
스레드 풀 = 워커 스레드(재사용) + 잡 큐(링버퍼) + mutex(큐 보호) + cond(대기/신호) + rwlock(엔진 보호)
```

- 워커는 잡이 없으면 CPU를 쓰지 않고 진짜로 잠듭니다
- SELECT는 동시에 여러 워커가 처리하고, INSERT만 단독으로 처리합니다
- 서버가 종료될 때 모든 워커를 안전하게 깨우고 리소스를 정리합니다
