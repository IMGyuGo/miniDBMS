# miniDBMS 4인 역할분담 개선안

> 기존 분담안(`4인_역할분담_및_진행계획.md`)을 검토한 결과를 반영한 수정본이다.
> 핵심 변경 사항은 **Role D 부하 분산**, **A→B 의존성 명시**, **공용 파일 합의 프로세스 구체화**, **단계별 마감 기준 추가**다.

---

## 1. 변경 요약

| 항목 | 기존안 | 개선안 |
|------|--------|--------|
| Role D 범위 | executor + HTTP + threadpool + 통합테스트 + README | executor + HTTP + 통합테스트 + README (threadpool 기초는 B로 이관) |
| Role B 범위 | index/파일 일관성 | index/파일 일관성 + **threadpool 기초 구현** |
| A→B 의존성 | 미명시 | 1단계에서 A가 인터페이스 먼저 확정 후 B 착수 명시 |
| 공용 파일 합의 | "짧게 합의 후 반영" 선언만 존재 | PR 리뷰 기반 합의 채널 및 머지 권한자 명시 |
| 단계별 마감 | 없음 | 각 단계에 완료 기준(Definition of Done) 추가 |

---

## 2. 역할분담 (수정본)

### 전체 요약

| 역할 | 담당 영역 | 주 소유 파일/디렉터리 |
|------|-----------|------------------------|
| Role A | B+Tree 저장 코어 | `src/bptree/**`, `include/bptree.h` |
| Role B | 인덱스/파일 일관성 + threadpool 기초 | `src/index/**`, `include/index_manager.h`, `src/threadpool/**` |
| Role C | SQL 입력/검증/계약 | `src/input/**`, `src/parser/**`, `src/schema/**` |
| Role D | 실행/HTTP 서버/통합 | `src/executor/**`, `src/main.c`, `src/server/**`, `src/http/**` |

---

### Role A — B+Tree 저장 코어

**핵심 목표:** 트리 정확성, range 탐색, 성능 측정 기반 안정화

**주요 책임:**
- B+Tree insert / search / range 동작 정확성 유지
- 노드 분할, 높이 변화, 중복 key 처리 정책 정리
- 성능 비교를 위한 tree height / I/O 측정 포인트 유지
- B+Tree 단위 테스트 작성

**권장 산출물:**
- `tests/test_bptree.c`
- B+Tree edge case 목록 문서
- 성능 지표 정리 (Role B와 공동 작성)

> **⚠️ 의존성 주의:** Role B는 A의 인터페이스(`include/bptree.h`) 확정 이후 본격 착수한다. 1단계 내에서 A가 함수 시그니처를 먼저 확정해야 한다.

---

### Role B — 인덱스/파일 일관성 + threadpool 기초

**핵심 목표:** `.dat` 파일과 인덱스 재구성 경로 안정화, 락이 필요한 구간 도출 후 threadpool 기초 구현

**주요 책임:**
- `data/{table}.dat` 와 인덱스 offset 경로 정리
- 시작 시 인덱스 재구성 전략 정리
- 빈 파일, 깨진 줄, 중복 키, 대량 데이터 예외 케이스 점검
- 저장 계층 기준으로 락 보호가 필요한 구간 문서화
- **threadpool 기초 구현** (worker thread 생성/종료, job queue 기본 구조)
  - 단, threadpool의 락 정책은 Role D와 합의 후 적용

**권장 산출물:**
- `tests/test_index.c`
- `tests/test_threadpool.c`
- 인덱스 rebuild 흐름 문서
- 저장 계층 보호 구간 체크리스트

> **역할 분리 이유:** B는 어차피 "어디에 락을 걸어야 하는가"를 정리해야 하므로, threadpool 기초 구현까지 자연스럽게 이어진다. D는 HTTP 레이어와 통합에 집중할 수 있게 된다.

---

### Role C — SQL 입력/검증/계약

**핵심 목표:** SQL 문자열을 안정적으로 AST/검증 결과로 바꾸고, 서버와 맞물릴 계약 구조 확정

**주요 책임:**
- lexer / parser / schema 안정화
- SQL 에러 메시지와 검증 실패 케이스 정리
- 서버 request/response 계약 초안 설계 (Role D보다 먼저 작성)
- 필요 시 다중 statement 정책 정리

**권장 산출물:**
- `tests/test_parser.c`
- `tests/test_schema.c`
- 에러 코드 및 결과 구조 초안 (`docs/api_contract.md`)
- API 예제 SQL 세트

> **추가 역할:** Role D가 HTTP 서버를 구현하기 전에 request/response 계약 문서를 먼저 완성하는 리드를 맡는다.

---

### Role D — 실행/HTTP 서버/통합

**핵심 목표:** 기존 엔진을 공용 서비스로 묶고, Role B의 threadpool 위에 HTTP 서버 MVP 연결

**주요 책임:**
- CLI 전용 실행 흐름을 공용 서비스 함수로 정리
- executor 결과를 서버 응답에 실을 수 있는 형태로 연결
- 신규 `src/server`, `src/http` 구현
- 동시성 모델 설계 (락 정책은 Role B와 합의)
- 통합 테스트 및 시연 가능한 실행 경로 정리

**1차 MVP 목표:**
- 공용 실행 진입 함수 1개 확정
- `POST /query` 하나만 제공하는 최소 HTTP 서버
- 응답 구조: `{ rows, elapsed_ms, error }`
- 락: 1차 버전은 global mutex 또는 단순 rwlock으로 안전성 우선

**권장 산출물:**
- `tests/test_executor.c`
- `tests/test_api.c`
- `tests/test_concurrency.c`
- README 실행 예시 초안
- 데모용 end-to-end 시나리오

## 3. 공용 파일 합의 프로세스 (신규)

기존안에서 "짧게 합의 후 반영"으로만 선언된 부분을 구체화한다.

### 공용 파일 목록

| 파일 | 머지 권한자 | 합의 방법 |
|------|------------|-----------|
| `include/interface.h` | Role C (계약 리드) | PR 리뷰 — 4인 중 2인 이상 승인 필요 |
| `Makefile` | Role D | PR 리뷰 — 변경 이유 코멘트 필수 |
| `include/db_service.h` (신규) | Role D | PR 리뷰 — Role C와 사전 합의 후 오픈 |
| `include/server_api.h` (신규) | Role D | PR 리뷰 — Role C가 계약 초안 먼저 작성 |

### 운영 규칙

1. 공용 파일 변경은 반드시 별도 PR로 올린다. 구현 PR에 끼워 넣지 않는다.
2. PR을 열기 전에 카카오톡/디스코드 등 팀 채널에 변경 의도를 먼저 공유한다.
3. 24시간 이내 리뷰가 없으면 작성자가 직접 다시 알림을 보낸다.
4. 긴급 수정이라도 혼자 머지하지 않는다. 최소 1인 이상 확인 후 머지한다.

## 4. 단계별 진행 순서 및 완료 기준 (수정본)

### 1단계. 공용 계약 고정

**목표:** 구현보다 "붙는 면"을 먼저 맞춘다.

| 역할 | 할 일 |
|------|--------|
| Role A | `include/bptree.h` 함수 시그니처 초안 확정 및 공유 |
| Role B | 인덱스 rebuild 흐름 및 락 보호 구간 문서 초안 작성 |
| Role C | request/response/에러 구조 초안(`docs/api_contract.md`) 작성 |
| Role D | 공용 실행 진입 함수 시그니처 초안 작성 |

**✅ 완료 기준 (이 4가지가 확정되어야 2단계 진입):**
- [ ] 서버가 엔진을 호출하는 함수 시그니처 1개 이상 확정
- [ ] 결과/에러 구조체 정의 확정
- [ ] 락 보호 구간 목록 초안 확정
- [ ] 첫 MVP API 범위 확정 (`POST /query` 1개)

---

### 2단계. 병렬 구현

**목표:** 각자 소유 모듈 구현 및 단위 테스트 작성

| 역할 | 할 일 |
|------|--------|
| Role A | B+Tree 안정화 + `tests/test_bptree.c` |
| Role B | index/path/rebuild 안정화 + threadpool 기초 + 단위 테스트 |
| Role C | parser/schema/에러 구조 정리 + 단위 테스트 |
| Role D | 공용 실행 서비스 함수 + HTTP 서버 MVP + 큐 연결 |

**✅ 완료 기준:**
- [ ] 각 역할 단위 테스트 `make test` 통과
- [ ] `POST /query` 로컬에서 동작 확인
- [ ] threadpool 기본 job enqueue/dequeue 동작 확인

---

### 3단계. 통합

**목표:** 모듈 간 연결 검증 및 일관성 점검

| 조합 | 확인 항목 |
|------|-----------|
| Role C + Role D | 요청/응답 계약과 실제 서버 연결 일치 여부 |
| Role B + Role D | 락 범위와 파일/인덱스 일관성 |
| Role A + Role B | 성능 수치와 인덱스 활용 경로 |
| 전원 | 통합 테스트 + 문서 갱신 |

**✅ 완료 기준:**
- [ ] end-to-end 통합 테스트 시나리오 1개 이상 통과
- [ ] 동시 요청 기본 시나리오에서 데이터 손상 없음 확인
- [ ] 성능 비교 수치 1차 측정 완료

## 7. 개선안 요약

이 문서의 핵심 변경은 아래 4가지다.

1. **Role D 부하 분산** — threadpool 기초 구현을 Role B로 이관해서 D가 HTTP + 통합에 집중할 수 있게 한다.
2. **A→B 의존성 명시** — 1단계에서 A의 인터페이스 확정이 B 착수의 전제 조건임을 팀 전체가 인지한다.
3. **공용 파일 합의 프로세스 구체화** — PR 리뷰 기반 합의 채널, 머지 권한자, 24시간 리뷰 룰을 명시한다.
4. **단계별 완료 기준 추가** — 각 단계에 체크리스트를 붙여서 다음 단계로의 진입 기준을 명확히 한다.
