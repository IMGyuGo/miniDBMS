# miniDBMS AI Convention

## 1. 목적

이 문서는 AI 에이전트가 `miniDBMS` 저장소에서 작업할 때 따라야 하는 공통 작업 규칙을 정리한 문서다.

이 문서의 목적은 아래 3가지다.

1. 역할별 파일 침범을 막는다.
2. 공통 인터페이스 계약을 함부로 바꾸지 않게 한다.
3. API 서버 확장 시 메시지 계층과 런타임 계층이 섞이지 않도록 판단 기준을 고정한다.

이 문서는 구현 문서가 아니라 작업 규약 문서다.

---

## 2. 읽는 순서

AI는 작업 전에 아래 순서로 문서를 확인한다.

1. `AGENT.md`
2. `docs/4인_역할분담_및_진행계획.md`
3. `docs/roles/README.md`
4. 자기 역할에 해당하는 `docs/roles/ROLE_*.md`
5. 관련 공통 헤더
   - `include/interface.h`
   - `include/bptree.h`
   - `include/index_manager.h`
   - `include/api_contract.h`
   - `include/db_service.h`

---

## 3. 기본 원칙

1. 이미 있는 엔진 구조를 버리지 않는다.
2. 역할별 1차 소유 범위를 존중한다.
3. 다른 역할 파일을 직접 고쳐서 빠르게 해결하지 않는다.
4. 공통 인터페이스는 필요할 때만, 합의 후 바꾼다.
5. 새 서버 기능은 계약 계층과 런타임 계층을 섞지 않는다.
6. 테스트는 구현과 같이 올린다.

---

## 4. 역할별 수정 범위

> ⚠️ **AI 경고 — 이 표는 작업 전 반드시 확인해야 하는 소유권 계약이다.**
> 자기 역할에 해당하지 않는 파일을 수정하는 것은 이 규약의 핵심 위반이다.
> "빠른 해결"을 이유로 타 역할 파일을 수정하는 것은 어떤 경우에도 허용되지 않는다.
> 필요한 변경이 있으면 공통 헤더 변경 제안 또는 주석 요청으로 처리한다.

| 역할 | 수정 가능 범위 | 절대 수정 금지 범위 |
|------|----------------|---------------------|
| Role A | `include/bptree.h`, `src/bptree/**`, `tests/test_bptree.c` | `src/index/**`, `src/executor/**`, `src/http/**` |
| Role B | `include/index_manager.h`, `src/index/**`, `tests/test_index.c` | `src/bptree/**`, `src/server/**`, `src/threadpool/**` |
| Role C | `src/input/**`, `src/parser/**`, `src/schema/**`, `src/http/**`, `include/api_contract.h`, `tests/test_parser.c`, `tests/test_schema.c`, `tests/test_http.c` | `src/server/**`, `src/threadpool/**`, `src/service/**` |
| Role D | `src/executor/**`, `src/main.c`, `src/server/**`, `src/threadpool/**`, `src/service/**`, `tests/test_executor.c`, `tests/test_threadpool.c`, `tests/test_api.c`, `tests/test_concurrency.c` | `src/http/**`, `src/parser/**`, `include/api_contract.h` 독단 수정 |

공통 합의 파일 (단독 수정 절대 금지):

- `AGENT.md`
- `docs/4인_역할분담_및_진행계획.md`
- `docs/ai_convention.md`
- `docs/roles/**`
- `include/interface.h`
- `include/db_service.h`
- `include/server_api.h`
- `Makefile`
- `scripts/check_ownership.sh`

설명:

- `include/api_contract.h` 는 Role C 1차 소유지만 Role D가 런타임에서 사용하므로 변경 시 Role D 확인이 필요하다.
- `README.md`, `samples/**` 같은 공통 자료는 역할 작업에서 함께 갱신할 수 있지만, 규약 문서와 공용 계약 문서는 합의 후 수정한다.

## 4-1. 역할 간 의존관계

아래는 역할 간 실제 의존 방향이다. AI는 이 방향을 기준으로 "내가 바꾸는 것이 누구에게 영향을 주는가"를 먼저 확인한다.

```text
Role A (bptree)
    └─▶ Role B (index_manager) 가 bptree.h API를 소비한다
            └─▶ Role D (executor/service) 가 index_manager.h API를 소비한다
                    └─▶ Role C (http) 가 api_contract.h + db_service.h 계약을 소비한다

Role C (api_contract.h / db_service.h 계약 설계)
    └─▶ Role D (service/server) 가 이 계약을 구현하고 호출한다
```

핵심 의존 규칙:

- Role A의 `bptree.h` API가 바뀌면 Role B에게 반드시 알린다.
- Role B의 `index_manager.h` API가 바뀌면 Role D에게 반드시 알린다.
- Role C의 `api_contract.h` / `db_service.h` 가 바뀌면 Role D에게 반드시 알린다.
- Role D가 `db_service_execute_sql` 시그니처를 바꾸면 Role C와 함께 검토한다.
- 의존 방향 반대로 (예: Role D가 Role A 코드를 직접 수정하는 것) 올라가는 수정은 금지한다.

각 역할의 선행 완료 조건:

| 역할 | 내가 시작하려면 먼저 완료되어야 하는 것 |
|------|----------------------------------------|
| Role B | Role A의 `bptree.h` API가 확정되어야 한다 |
| Role D | Role C의 `api_contract.h` 초안과 Role B의 `index_manager.h` API가 확정되어야 한다 |
| Role C | 독립적으로 시작 가능. 단 `db_service.h` 계약은 Role D와 합의 후 확정한다 |
| Role A | 독립적으로 시작 가능 |

## 4-2. 역할별 현재 할 일 요약

AI는 작업 시작 전에 아래 표를 확인하고, 현재 어느 단계인지 파악한다.

| 역할 | 1단계 (계약 먼저) | 2단계 (병렬 구현) | 3단계 (통합) |
|------|------------------|------------------|--------------|
| Role A | B+Tree 공개 API 확정, edge case 정리 | split/insert/range 안정화, `tests/test_bptree.c` 작성 | Role B와 성능 수치 공동 검증 |
| Role B | 인덱스 rebuild 흐름, 락 보호 구간 정리 | index path 안정화, `tests/test_index.c` 작성 | Role D와 락 범위 맞추기 |
| Role C | `api_contract.h` 초안, HTTP 요청/응답 형식 확정 | parser/schema/HTTP 메시지 계층 정리, `tests/test_http.c` 등 작성 | Role D와 요청/응답 계약 점검 |
| Role D | `db_service_execute_sql` 시그니처 확정 | service/server/threadpool 구현, `POST /query` MVP | Role C + Role B와 통합 검증 |

## 5. 공유 인터페이스 계약

이 섹션은 AI가 함수 시그니처를 추측하지 않도록 실제 계약을 코드 블록으로 고정한 것이다.
아래 시그니처를 기준으로 동작을 가정한다. 실제 헤더와 다를 경우 헤더가 우선한다.

### 5-1. 엔진 핵심 실행 경계 (`include/db_service.h`)

```c
/* 공용 실행 진입점 — 서버/HTTP 계층은 이 함수만 호출한다 */
DBServiceStatus db_service_execute_sql(const char *sql,
                                       const DBServiceOptions *opts,
                                       DBServiceResult *out);

/* 결과 구조 생명주기 */
void db_service_options_init(DBServiceOptions *opts);
void db_service_result_init(DBServiceResult *result);
void db_service_result_reset(DBServiceResult *result);
void db_service_result_free(DBServiceResult *result);

/* 실행 결과 구조 */
typedef struct {
    DBServiceStatus status;       /* SQL_OK(0) or 오류 코드 */
    StmtType        stmt_type;
    int             rows_affected;
    int             has_result_set;
    int             has_profile;
    ResultSet      *result_set;   /* SELECT 결과, db_service_result_free()로 해제 */
    DBServiceProfile profile;
    char            message[256]; /* 서비스 계층 오류 메시지 */
} DBServiceResult;

/* 실행 옵션 */
typedef struct {
    int force_linear;   /* 1이면 인덱스 대신 선형 경로 강제 */
    int compare;        /* 비교 실행 요청 */
    int emit_log;       /* stderr 진단 로그 출력 여부 */
    int include_profile;/* 결과에 profile 메타데이터 포함 여부 */
} DBServiceOptions;
```

### 5-2. B+Tree 공개 API (`include/bptree.h`) — Role A 소유

```c
/* 트리 생명주기 */
BPTree *bptree_create(int order);
void    bptree_destroy(BPTree *tree);

/* 핵심 연산 */
int  bptree_insert(BPTree *tree, int key, long value);
long bptree_search(BPTree *tree, int key);        /* 없으면 -1 */

/* 범위 탐색 */
int   bptree_range(BPTree *tree, int from, int to, long *out, int max_count);
long *bptree_range_alloc(BPTree *tree, int from, int to, int *out_count); /* caller가 free */

/* 진단 */
int  bptree_height(BPTree *tree);
int  bptree_last_io(BPTree *tree);
void bptree_print(BPTree *tree);
```

> Role B / Role D는 위 API만 사용한다. 트리 내부 구조(`BPNode` 등)를 직접 접근하면 안 된다.

### 5-3. 인덱스 공개 API (`include/index_manager.h`) — Role B 소유

```c
/* 초기화/정리 */
int  index_init(const char *table, int order_id, int order_age);
void index_cleanup(void);

/* id 인덱스 */
int  index_insert_id(const char *table, int id, long offset);
long index_search_id(const char *table, int id);
int  index_range_id(const char *table, int from, int to, long *offsets, int max);
long *index_range_id_alloc(const char *table, int from, int to, int *out_count);

/* age 인덱스 */
int  index_insert_age(const char *table, int age, long offset);
int  index_range_age(const char *table, int from, int to, long *offsets, int max);
long *index_range_age_alloc(const char *table, int from, int to, int *out_count);

/* 진단 */
int  index_height_id(const char *table);
int  index_height_age(const char *table);
void index_reset_io_stats(const char *table);
int  index_last_io_id(const char *table);
int  index_last_io_age(const char *table);
```

> Role D(executor/service)는 인덱스 사용 시 위 API만 쓴다. `src/index/**` 내부 구현을 직접 호출하면 안 된다.

### 5-4. HTTP 메시지 계약 (`include/api_contract.h`) — Role C 소유

```c
/* 상수 */
#define API_ROUTE_QUERY   "/query"
#define API_ROUTE_HEALTH  "/health"
#define API_SQL_MAX_LEN      4096
#define API_ERROR_MAX_LEN     256
#define API_REQUEST_ID_MAX     64

/* MVP 기준: 한 요청에 하나의 SQL statement만 허용 */
/* multi-statement는 합의 전까지 거절한다 */

/* 애플리케이션 레벨 오류 코드 */
typedef enum {
    API_CODE_OK = 0,
    API_CODE_BAD_REQUEST,
    API_CODE_UNSUPPORTED_ROUTE,
    API_CODE_UNSUPPORTED_METHOD,
    API_CODE_PARSE_ERROR,
    API_CODE_SCHEMA_ERROR,
    API_CODE_EXEC_ERROR,
    API_CODE_QUEUE_FULL,
    API_CODE_INTERNAL_ERROR
} ApiCode;

/* 쿼리 실행 옵션 (transport 계층에서 수신, service 계층이 적용 판단) */
typedef struct {
    int force_linear;    /* 1이면 인덱스 대신 선형 경로 강제 */
    int compare;         /* 비교 실행 요청, MVP에서는 미지원 처리 가능 */
    int include_profile; /* 응답에 실행 프로파일 포함 요청 */
} ApiQueryOptions;

/* HTTP parser가 body를 해석한 뒤 넘기는 request-local 구조 */
typedef struct {
    char            request_id[API_REQUEST_ID_MAX];
    ApiMethod       method;
    ApiRouteKind    route;
    ApiQueryOptions options;
    char            sql[API_SQL_MAX_LEN];  /* NUL-terminated UTF-8 */
} ApiQueryRequest;

/* transport metadata + 오류/상태 표현 (실제 row payload는 serializer가 변환) */
typedef struct {
    int      http_status;
    ApiCode  code;
    int      ok;
    int      has_payload;
    int      row_count;
    int      rows_affected;
    char     error[API_ERROR_MAX_LEN]; /* HTTP 응답 오류 문자열 */
} ApiResponseMeta;

/* 오류 응답 문자열: ApiResponseMeta.error */
/* 서비스 계층 오류 메시지: DBServiceResult.message */
/* HTTP status 매핑: ApiCode 기준으로 transport 계층이 처리 */
```

## 5-5. 인터페이스 배치 판단 기준

### `include/interface.h`

- SQL 엔진 공통 계약만 둔다.
- 토큰, AST, 스키마, `ResultSet` 같은 엔진 도메인 구조만 유지한다.
- HTTP 헤더, status code, 소켓 fd, threadpool 상태는 넣지 않는다.

### `include/api_contract.h`

- HTTP 요청/응답 메시지 계약만 둔다.
- route, method, request option, application-level error code를 둔다.
- 엔진 내부 구조(`AST`, `ResultSet`)는 직접 넣지 않는다.
- `ApiQueryRequest`, `ApiResponseMeta` 같은 transport 구조는 여기서 관리한다.

### `include/db_service.h`

- 서버가 엔진을 호출하는 공용 서비스 경계를 둔다.
- transport 비의존적이어야 한다.
- parser/executor를 직접 우회 호출하지 않도록 service API로 고정한다.
- 현재 핵심 진입점은 `db_service_execute_sql(const char *sql, const DBServiceOptions *opts, DBServiceResult *out)` 이다.
- 서비스 계층 오류 메시지는 `DBServiceResult.message`, HTTP 응답 오류 문자열은 `ApiResponseMeta.error` 에 둔다.

### `include/bptree.h`

- Role A 공개 API다.
- Role B, Role D는 이 헤더를 통해서만 트리 기능을 사용한다.
- 트리 내부 구조를 외부 계층이 직접 가정하면 안 된다.

### `include/index_manager.h`

- Role B 공개 API다.
- Role D는 `index_*` API를 통해서만 인덱스를 사용한다.
- executor/service가 저장 계층 내부 구현을 직접 만지면 안 된다.

## 6. API 서버 확장 규칙

API 서버를 추가할 때 AI는 아래 계층 분리를 지킨다.

런타임 계층:

- `src/server/**` : accept loop / connection lifecycle / 종료 흐름
- `src/threadpool/**` : worker / queue / 스케줄링

요청 처리 경로:

```text
HTTP request
  -> src/http/*       : message parsing / response formatting
  -> src/service/*    : engine call boundary
  -> src/executor/*   : SQL execution
  -> src/index/*      : storage index
  -> src/bptree/*     : tree core
```

규칙:

1. `src/http/**` 는 메시지 규약만 다룬다.
2. `src/server/**` 는 소켓 accept loop, 연결 관리, 종료 흐름을 맡는다.
3. `src/threadpool/**` 는 worker lifecycle과 queue만 맡는다.
4. `src/service/**` 는 transport와 executor 사이의 단일 공용 실행 경계다.
5. `src/http/**` 와 `src/server/**` 의 책임을 한 파일에 섞지 않는다.

---

## 7. 역할 충돌 방지 규칙

1. Role C는 `src/http/**` 만 맡고 `src/server/**`, `src/threadpool/**`, `src/service/**` 를 직접 수정하지 않는다.
2. Role D는 메시지 규약을 바꾸기 위해 `src/http/**` 또는 `include/api_contract.h` 를 직접 수정하지 않는다.
3. Role B는 인덱스 문제를 해결하려고 `src/bptree/**` 를 직접 수정하지 않는다.
4. Role A는 트리 문제를 index/executor 우회 수정으로 덮지 않는다.
5. 공통 계약이 부족하면 남의 소유 파일을 건드리지 말고 공통 헤더 변경을 제안한다.

---

## 8. 새 필드/새 구조체 추가 규칙

AI가 새 필드나 새 구조체를 추가하려고 할 때는 아래 질문을 먼저 확인한다.

1. 이 값이 SQL 엔진 의미 자체에 필요한가?
2. transport/runtime 정보가 아니라 parser/schema/executor 공통 의미인가?
3. 요청 단위 데이터인가, 런타임 공유 상태인가?
4. free 책임이 명확한가?
5. 역할 경계가 흐려지지 않는가?

판단 기준:

- 엔진 공통 의미면 `include/interface.h`
- HTTP 요청/응답 의미면 `include/api_contract.h`
- 엔진 호출 결과/프로파일 의미면 `include/db_service.h`
- 저장/트리 의미면 해당 역할 헤더
- 런타임 공유 상태면 `src/server/**` 또는 `src/threadpool/**` 내부 구조

## 8-1. 코드 스타일 규칙

AI가 코드를 작성할 때 아래 스타일을 따른다. 기존 파일과 충돌하면 기존 파일 스타일을 따른다.

**네이밍 규칙:**

- 함수명: `snake_case` (예: `db_service_execute_sql`, `bptree_insert`)
- 구조체/열거형: `PascalCase` (예: `DBServiceResult`, `TokenType`)
- 매크로/상수: `UPPER_SNAKE_CASE` (예: `SQL_OK`, `API_SQL_MAX_LEN`)
- 지역 변수: `snake_case` (예: `row_count`, `elapsed_ms`)
- 파일명: `snake_case.c` / `snake_case.h`

**들여쓰기/포맷:**

- 들여쓰기는 스페이스 4칸을 기본으로 한다.
- 중괄호는 K&R 스타일 (`if (...) {` 형태).
- 한 줄 최대 길이는 100자 이내를 권장한다.

**헤더 파일:**

- 모든 헤더는 `#ifndef` / `#define` / `#endif` 가드를 사용한다.
- 헤더에는 선언(declaration)만 두고 구현(definition)은 `.c` 파일에 둔다.
- 메모리 소유권이 있는 포인터는 반드시 주석으로 해제 책임자를 명시한다.

**주석:**

- 공개 API 함수는 함수 위에 계약 주석을 단다 (전제조건, 반환값, 소유권).
- `TODO` / `FIXME` 는 역할 이니셜을 포함한다 (예: `/* TODO(RoleD): ... */`).

---

## 9. 코드/에러 처리 규칙

1. 엔진 내부 정수 반환 함수는 `SQL_OK(0)` / `SQL_ERR(-1)` 패턴을 유지한다.
2. 포인터 반환 함수는 성공 시 유효 포인터, 실패 시 `NULL` 을 반환한다.
3. 결과 데이터는 `stdout`, 진단과 오류는 `stderr` 로 분리한다.
4. 서비스 계층은 `DBServiceResult.message` 로 오류를 전달하고, transport 계층은 이를 `ApiResponseMeta.error` 로 매핑한다.
5. `ExecResult` 같은 별도 타입을 가정하지 않는다. 실제 계약은 `DBServiceResult` 와 `ApiResponseMeta` 가 기준이다.

---

## 10. 출력 및 I/O 규칙

1. 결과 row는 `stdout`
2. 타이밍/진단은 `stderr`
3. `data/*.dat` 접근 `fopen()`은 반드시 binary 모드를 사용한다 (`"rb"`, `"ab"`, `"wb"`). text 모드(`"r"`, `"a"`, `"w"`)는 Windows에서 `ftell` 오프셋 오류를 일으킨다.
4. `schema/users.schema` 는 임의 변경 금지
5. binary 모드 준수는 `scripts/check_binary_mode.sh` 로 자동 검증된다. 커밋 전 실행하거나 git hook에서 확인한다.

---

## 11. 테스트 규칙

1. 단위 테스트는 역할별 소유를 따른다.
2. 구현 파일을 수정하면 관련 테스트도 같이 수정한다.
3. 통합 테스트는 Role D가 리드하되, 계약 검증은 Role C가 같이 본다.
4. 성능 비교 자료는 Role A + Role B가 같이 정리한다.

---

## 12. 커밋 규칙

1. 한 커밋에는 가능한 한 한 역할의 소유 파일만 담는다.
2. 공통 파일이 섞이면 변경 이유가 분명해야 한다.
3. `scripts/check_ownership.sh` 기준을 우회하려고 파일을 섞지 않는다.
4. 커밋 전 `scripts/lint.sh` 로 컴파일 검증을 통과해야 한다 (`-Wall -Wextra` 기준).
5. `scripts/check_binary_mode.sh` 도 커밋 전 통과 여부를 확인한다.
6. `scripts/install_hooks.sh` 를 실행하면 위 검증이 git pre-commit hook으로 자동 등록된다.

## 13. 금지 예시

- HTTP 응답 형식을 바꾸려고 `src/executor/**` 에 JSON 직렬화 코드를 넣는 것
- 서버 성능 문제를 해결하려고 `src/http/**` 에 worker queue를 구현하는 것
- 인덱스 문제를 고치려고 `src/bptree/**` 내부 구조를 Role B가 직접 수정하는 것
- 공통 헤더를 빠르게 바꾸고 관련 역할에게 공유하지 않는 것

## 14. AI 행동 규약

이 섹션은 AI가 작업 중 반드시 따라야 하는 행동 원칙을 명령형으로 정리한 것이다.

> ⚠️ 이 섹션의 규칙은 다른 어떤 판단보다 우선한다.
> AI는 "빠른 해결"을 이유로 아래 규칙을 어기면 안 된다.

**작업 시작 전 체크리스트 — 아래를 순서대로 확인한다:**

1. 현재 내 역할이 무엇인지 확인한다.
2. 섹션 4 소유권 표에서 내가 수정 가능한 파일 목록을 확인한다.
3. 섹션 4-1 의존관계에서 내 변경이 누구에게 영향을 주는지 확인한다.
4. 수정할 파일이 공통 합의 파일인지 확인한다.

**담당 파일 외 수정 금지:**

- 자기 역할 소유 파일이 아닌 파일은 절대 직접 수정하지 않는다.
- 타 역할 파일에 버그가 있더라도 직접 고치지 않는다. 주석으로 문제를 표시하거나 공통 헤더 변경을 제안한다.
- "내가 잠깐만 수정하면 되는데"라는 판단이 드는 순간, 그것은 역할 침범이다.

**공통 파일 변경 금지:**

- `include/interface.h`, `include/db_service.h`, `AGENT.md`, `Makefile`, `scripts/check_ownership.sh` 는 단독으로 수정하지 않는다.
- 공통 파일을 바꾸고 싶으면 변경이 필요한 이유와 예상 영향을 주석 또는 별도 메모로 남기고 합의를 요청한다.

**함수 시그니처 / 구조체 임의 변경 금지:**

- 섹션 5의 공유 인터페이스 계약을 사전 합의 없이 변경하지 않는다.
- 공개 API 함수의 파라미터 타입, 반환 타입, 이름을 임의로 바꾸지 않는다.
- 새 필드/구조체가 필요하면 섹션 8의 판단 기준을 먼저 적용한다.

**새 파일 추가 규칙:**

- 새 파일은 자기 역할의 소유 디렉터리 안에 만든다.
- 새 헤더를 추가할 때 기존 공통 헤더와 역할이 겹치면 먼저 확인한다.
- 테스트 파일은 구현과 같은 커밋에 포함한다.

**금지 행동 — AI는 아래 행동을 절대 하지 않는다:**

- 다른 역할 파일을 "일단 고치고 나중에 알림" 방식으로 처리하는 것
- 공통 헤더를 단독으로 수정하고 관련 역할에 공유하지 않는 것
- `src/http/**` 에 실행 로직(executor 호출, 인덱스 접근)을 추가하는 것
- `src/executor/**` 에 HTTP 직렬화 / JSON 포맷 코드를 추가하는 것
- `src/server/**` 또는 `src/threadpool/**` 에 SQL 파싱 코드를 추가하는 것
- `src/bptree/**` 내부 구조를 Role B 또는 Role D가 직접 수정하는 것
- Role C가 서버 런타임 편의를 위해 `src/server/**`를 직접 수정하는 것
