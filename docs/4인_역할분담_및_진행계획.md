# miniDBMS 4인 역할분담 및 진행 계획

> 운영 기준 문서는 루트 `AGENT.md` 와 `docs/roles/*.md` 이고, 이 문서는 그 기준이 되는 분담 계획 문서다.

## 1. 문서 목적

이 문서는 현재 저장소 구조와 이미 존재하는 소유권 규칙을 기준으로, `miniDBMS` 프로젝트를 4명이 가장 적게 충돌하면서 병렬로 진행하기 위한 역할분담 재정비안이다.

핵심 목표는 아래 4가지다.

1. 이미 존재하는 `A/B/C/D` 모듈 소유권을 최대한 유지한다.
2. Role D에 과도하게 몰리던 신규 서버 계층 책임을 적절히 분산한다.
3. API 서버 구현 시 계약 계층과 런타임 계층을 분리해 충돌을 줄인다.
4. 테스트와 문서는 "별도 역할 한 명"에게 몰지 않고 각 모듈 책임에 포함시킨다.

---

## 2. 현재 저장소 상태 요약

현재 코드 기준으로 확인한 내용은 아래와 같다.

| 구분 | 현재 상태 |
|------|-----------|
| 엔진 진입점 | `src/main.c` 에 CLI 실행 흐름이 있다. |
| SQL 처리 | `src/input`, `src/parser`, `src/schema`, `src/executor` 가 연결되어 있다. |
| 인덱스 | `src/index/index_manager.c`, `src/bptree/bptree.c` 가 존재하고 executor와 연결되어 있다. |
| 지원 기능 | `INSERT`, `SELECT`, `WHERE col = value`, `WHERE col BETWEEN a AND b` 가 들어가 있다. |
| 스키마 | 현재 `schema/users.schema` 기준으로 `users` 테이블을 사용한다. |
| 데이터 저장 | `data/{table}.dat` 파일 기반이다. |
| 서버 계층 | `src/server`, `src/http`, `src/threadpool`, `src/service` 같은 API 서버 계층은 아직 없다. |
| 네트워크/동시성 | HTTP 처리, 요청 큐, 스레드 풀, 락 구조는 아직 없다. |
| 소유권 규칙 | `include/interface.h`, `scripts/check_ownership.sh` 에 이미 A/B/C/D 역할 구분이 들어가 있다. |
| 테스트 | `Makefile`에는 테스트 타깃이 있지만 현재 `tests` 소스는 아직 추가되지 않았다. |
| 문서 | `README.md`는 비어 있고, `docs/`에는 계획 문서가 막 들어오기 시작한 상태다. |

즉 지금 상태는 "CLI 기반 엔진은 이미 있고, 앞으로 서버 계층과 통합 테스트를 얹어야 하는 단계"로 보는 것이 가장 맞다.

---

## 3. 재분배 원칙

이번 역할 재분배는 아래 원칙을 따른다.

1. 기존 A/B/C/D 소유권 체계를 뒤집지 않는다.
2. 새로 생기는 코드도 반드시 1차 소유자를 둔다.
3. API 서버 관련 신규 코드는 계약/메시지 계층과 런타임 계층으로 나눠 병목을 줄인다.
4. 테스트는 구현자 책임 원칙으로 가져가고, 통합 테스트만 별도 리드가 묶는다.
5. 공용 파일은 적게 만들고, 만들어야 한다면 합의 후 수정한다.

이 원칙을 적용하면 가장 현실적인 4축은 아래와 같다.

- A: 저장 코어(B+Tree)
- B: 저장 경로(Index/Persistence)
- C: SQL 전단 + API 계약/HTTP 메시지
- D: 실행/CLI/서버 런타임

## 4. 권장 4인 역할분담

| 역할 | 담당 영역 | 주 소유 파일/디렉터리 | 핵심 목표 | 바로 시작할 일 |
|------|-----------|------------------------|-----------|----------------|
| Role A | B+Tree 저장 코어 | `src/bptree/**`, `include/bptree.h` | 트리 정확성, range 탐색, 성능 측정 기반 안정화 | split/insert/range edge case 점검, B+Tree 테스트 초안 작성 |
| Role B | 인덱스/파일 일관성 | `src/index/**`, `include/index_manager.h` | `.dat` 파일과 인덱스 재구성 경로 안정화 | `index_init` 점검, 재시작 시 rebuild 흐름 정리, index 테스트 초안 작성 |
| Role C | SQL 입력/검증/계약/HTTP 메시지 | `src/input/**`, `src/parser/**`, `src/schema/**`, 신규 `src/http/**`, 신규 `include/api_contract.h` | SQL 입력을 안정적으로 AST/검증 결과로 바꾸고, 서버와 맞물릴 계약과 메시지 계층 정리 | 에러 코드/결과 구조 초안, parser/schema 테스트 초안, HTTP 요청/응답 형식 초안 작성 |
| Role D | 실행/CLI/서버 런타임 | `src/executor/**`, `src/main.c`, 신규 `src/server/**`, `src/threadpool/**`, 신규 `src/service/**` | 기존 엔진을 공용 서비스로 묶고 API 서버 MVP까지 연결 | 공용 실행 진입 함수 설계, `POST /query` MVP, executor/threadpool/integration 테스트 초안 작성 |

이 재분배의 핵심은 "기존 소유 파일은 그대로 두고, 새 서버 계층은 C와 D가 계약과 런타임으로 나눠 맡는다"는 점이다.

---

## 5. 왜 이 분담이 더 적절한가

기존 문서처럼 역할을 완전히 새로 다시 나누면 보기에는 깔끔하지만, 현재 저장소에는 이미 아래 규칙이 들어가 있다.

- `include/interface.h` 에 A/B/C/D 모듈 경계가 적혀 있다.
- `scripts/check_ownership.sh` 가 소유자 혼합 커밋을 막고 있다.

따라서 가장 안전한 방향은 "역할명을 바꾸는 것"이 아니라 "기존 역할에 앞으로 할 일을 재배치하는 것"이다.

특히 아래 판단이 중요하다.

- B+Tree와 Index는 이미 물리적으로 파일이 분리되어 있으므로 A/B 분리가 자연스럽다.
- SQL 입력과 검증은 C가 계속 맡아야 인터페이스가 흔들리지 않는다.
- HTTP 메시지 파싱과 API 계약은 C가 맡고, 소켓/큐/워커 스레드 같은 서버 런타임은 D가 맡아야 서버 계층 내부 충돌이 줄어든다.
- 서버는 결국 executor와 CLI 경계를 공용 서비스로 바꿔야 하므로 D가 실행 통합을 맡는 편이 가장 자연스럽다.
- 테스트와 문서는 전담 한 명보다 각 역할이 자기 모듈 검증까지 책임지는 쪽이 실제 완성도가 높다.

## 6. 각 역할의 상세 책임

### Role A. B+Tree 저장 코어 담당

이 역할은 프로젝트의 가장 낮은 레벨 자료구조를 맡는다.

주요 책임:

- B+Tree insert/search/range 동작 정확성 유지
- 노드 분할, 높이 변화, 중복 key 처리 정책 정리
- 성능 비교를 위한 tree height / I/O 측정 포인트 유지
- B+Tree 단위 테스트 작성

권장 산출물:

- `tests/test_bptree.c`
- B+Tree edge case 목록 문서
- 성능 비교 시 확인할 지표 정리

### Role B. 인덱스/파일 일관성 담당

이 역할은 트리 자체보다 "트리를 실제 데이터 파일과 어떻게 연결하느냐"를 맡는다.

주요 책임:

- `data/{table}.dat` 와 인덱스 offset 경로 정리
- 시작 시 인덱스 재구성 전략 정리
- 빈 파일, 깨진 줄, 중복 키, 대량 데이터 같은 예외 케이스 점검
- 저장 계층 기준으로 어떤 구간이 락 보호 대상인지 정리
- 인덱스 단위 테스트 작성

권장 산출물:

- `tests/test_index.c`
- 인덱스 rebuild 흐름 문서
- 저장 계층 보호 구간 체크리스트

주의:

이 역할의 "복구"는 WAL 수준의 crash recovery가 아니라, 현재 구조 기준으로 `.dat` 재스캔을 통한 인덱스 재구성에 더 가깝다.

### Role C. SQL 입력/검증/계약/HTTP 메시지 담당

이 역할은 SQL 문자열을 안정적으로 해석해서 실행 가능한 구조로 바꾸는 전단과, 서버가 주고받을 메시지 계약을 맡는다.

주요 책임:

- lexer/parser/schema 안정화
- SQL 에러 메시지와 검증 실패 케이스 정리
- 서버가 받는 요청과 엔진이 반환하는 결과 구조의 초안 설계
- HTTP request parsing 과 response formatting 계층 초안 작성
- 필요 시 다중 statement 정책 정리
- parser/schema/http 단위 테스트 작성

권장 산출물:

- `tests/test_parser.c`
- `tests/test_schema.c`
- `tests/test_http.c`
- 에러 코드/결과 구조 초안
- API 예제 SQL 세트

추가 역할:

Role D가 서버를 구현할 때 쓸 수 있도록 request/response 계약 문서를 먼저 만드는 리드를 맡는다.

### Role D. 실행/CLI/서버 런타임 담당

이 역할은 현재 executor와 `main.c`를 이미 소유하고 있으므로, 앞으로 추가될 서버 런타임과 공용 실행 서비스의 1차 소유자도 함께 맡는 것이 가장 자연스럽다.

주요 책임:

- CLI 전용 실행 흐름을 공용 서비스 함수로 정리
- executor 결과를 서버 응답에 실을 수 있는 형태로 연결
- 신규 `src/server`, `src/threadpool`, `src/service` 구현
- 초기 동시성 모델 설계
- 통합 테스트와 시연 가능한 실행 경로 정리

권장 1차 목표:

- 공용 실행 진입 함수 1개 확정
- `POST /query` 하나만 제공하는 최소 서버 MVP
- 응답은 `rows`, `elapsed_ms`, `error` 중심의 단순 구조
- 락은 1차 버전에서 global mutex 또는 단순 rwlock 기준으로 안전성 우선
- 서버는 parser/executor 내부 함수를 직접 흩어 호출하지 않고 공용 실행 함수 하나로만 엔진을 호출

권장 산출물:

- `tests/test_executor.c`
- 이후 `tests/test_threadpool.c`, `tests/test_api.c`, `tests/test_concurrency.c`
- README 실행 예시 초안
- 데모용 end-to-end 시나리오

## 7. 테스트와 문서 책임 재배치

이 문서에서 가장 크게 바꾸는 점은 "테스트/문서 전담 Role 1명" 구조를 없애는 것이다.

그 대신 아래처럼 가져가는 것이 더 현실적이다.

| 구분 | 1차 책임자 | 설명 |
|------|------------|------|
| B+Tree 단위 테스트 | Role A | 자기 모듈 테스트는 구현자가 바로 붙인다. |
| Index 단위 테스트 | Role B | rebuild, offset, 예외 케이스 포함 |
| Parser/Schema 테스트 | Role C | 입력 검증, 오류 케이스 포함 |
| HTTP 메시지 테스트 | Role C | 요청 파싱, 응답 포맷 검증 |
| Executor/서버 통합 테스트 | Role D | 실제 end-to-end 동작 검증 |
| README / 실행 가이드 | Role D 리드, 전원 보강 | 서버 실행 흐름까지 묶어야 하므로 D가 초안 작성 |
| docs 보강 | 각 역할별 작성 | 자기 기능 바뀌면 같은 작업에서 문서도 함께 갱신 |
| 성능 비교 자료 | Role A + Role B | 트리와 인덱스 관점에서 공동 정리 |
| API 명세 예시 | Role C + Role D | 계약은 C, 실제 실행 예시는 D |

이 구조의 장점은 테스트가 마지막에 밀리지 않는다는 점이다.

## 8. 파일 소유 범위 제안

현재 저장소 규칙과 충돌하지 않도록 우선 소유 범위는 아래처럼 정리한다.

| 역할 | 우선 소유 범위 |
|------|----------------|
| Role A | `src/bptree/**`, `include/bptree.h` |
| Role B | `src/index/**`, `include/index_manager.h` |
| Role C | `src/input/**`, `src/parser/**`, `src/schema/**`, 신규 `src/http/**`, 신규 `include/api_contract.h` |
| Role D | `src/executor/**`, `src/main.c`, 신규 `src/server/**`, `src/threadpool/**`, 신규 `src/service/**` |

공통 합의 파일 후보:

- `include/interface.h`
- `Makefile`
- `docs/4인_역할분담_및_진행계획.md`
- `docs/ai_convention.md`
- 새로 추가될 공용 헤더 (`include/db_service.h`, `include/server_api.h` 등)

운영 규칙:

1. 소유 파일은 각 역할이 1차 책임을 진다.
2. 공통 파일은 함수 시그니처와 구조체를 먼저 합의하고 수정한다.
3. 단위 테스트 파일은 역할 소유를 따른다.
   - `tests/test_bptree.c` -> Role A
   - `tests/test_index.c` -> Role B
   - `tests/test_parser.c`, `tests/test_schema.c`, `tests/test_http.c` -> Role C
   - `tests/test_executor.c`, `tests/test_threadpool.c`, `tests/test_api.c`, `tests/test_concurrency.c` -> Role D
4. `README.md`, `samples/**` 같은 공통 자료는 역할 작업에서 함께 갱신할 수 있다. 다만 역할 규약 문서인 `AGENT.md`, `docs/roles/**`, `docs/4인_역할분담_및_진행계획.md`, `docs/ai_convention.md`, `scripts/check_ownership.sh`, 공용 계약 문서는 팀 합의 후 수정한다.
5. 가능하면 한 커밋에는 한 역할의 소유 파일과 공통 파일만 함께 담는다.

## 9. 실제 진행 순서 추천

### 1단계. 공용 계약 먼저 고정

이 단계에서는 구현보다 "붙는 면"을 먼저 맞춘다.

- Role A: 트리 측면에서 보장해야 할 연산/성능 지표 정리
- Role B: 인덱스 rebuild 및 보호 구간 정리
- Role C: 요청/오류/결과 구조와 HTTP 메시지 형식 초안 작성
- Role D: 엔진 공용 실행 함수 초안 작성

이 단계가 끝나면 아래 5가지는 최소 확정되어 있어야 한다.

- 서버가 엔진을 어떤 함수로 호출할지
- 결과와 에러를 어떤 구조로 받을지
- HTTP body 를 어떤 형식으로 주고받을지
- 어떤 구간에 락을 걸지
- 첫 번째 MVP API가 어디까지인지

### 2단계. 병렬 구현

- Role A: B+Tree 안정화와 단위 테스트
- Role B: index/path/rebuild 안정화와 단위 테스트
- Role C: parser/schema/에러 구조/HTTP 메시지 계층 정리와 테스트
- Role D: 공용 실행 서비스, HTTP 서버 런타임, 큐, 워커 스레드 구현

### 3단계. 통합

- Role C + Role D: 요청/응답 계약과 실제 서버 연결 점검
- Role B + Role D: 락 범위와 파일/인덱스 일관성 점검
- Role A + Role B: 성능 수치와 인덱스 활용 경로 검증
- 전원: 통합 테스트와 문서 갱신

### 4단계. 발표 준비

- Role D: README와 데모 흐름 초안 정리
- Role C: API 설명 자료 정리
- Role A + Role B: 성능 비교 포인트 정리
- 전원: 자기 담당 모듈 설명 1페이지씩 준비

## 10. 역할 간 인터페이스 규칙

작업 충돌을 줄이려면 아래 규칙을 미리 두는 것이 좋다.

1. Role D는 엔진 내부를 임의로 우회하지 말고 공용 실행 경계를 통해 서버를 붙인다.
2. Role B는 락 코드를 저장 계층 곳곳에 먼저 넣기보다, 보호가 필요한 구간을 문서화해서 Role D와 합의한다.
3. Role C는 새 기능을 넣을 때 파서 구조뿐 아니라 오류 메시지, 결과 구조, HTTP 메시지 규약까지 같이 정의한다.
4. Role C는 HTTP 메시지 파싱/포맷 계층만 맡고, 소켓 accept loop 와 worker runtime 은 Role D가 맡는다.
5. 각 역할은 자기 모듈 테스트를 같은 작업 안에서 같이 올린다.
6. `include/interface.h`, `Makefile`, 공용 헤더 변경은 짧게라도 합의 후 반영한다.
7. 소유권 스크립트와 충돌하지 않도록, 여러 역할 소유 파일을 한 커밋에 섞는 일은 피한다.

## 11. 이 분담안의 장점

이 구조로 나누면 장점이 분명하다.

- 현재 저장소의 A/B/C/D 소유권 규칙과 충돌하지 않는다.
- 새 서버 계층을 억지로 한 사람에게 몰지 않고 C와 D가 계약/런타임으로 나눠 맡을 수 있다.
- 테스트가 특정 한 명에게 몰리지 않고 구현과 같이 진행된다.
- A/B/C는 기존 모듈을 안정화하면서도 자기 영역의 산출물이 분명하다.
- D는 실행 통합과 서버 런타임에 집중할 수 있어 병목이 줄어든다.
- 공용 파일 충돌이 `include/`와 `Makefile` 정도로 줄어든다.
- 발표 준비도 "문서 담당 한 명"이 아니라 각자 근거를 가지고 준비할 수 있다.

## 12. 최종 제안 요약

현재 저장소 기준으로 가장 현실적인 4인 분담은 아래 한 줄로 정리된다.

> Role A는 B+Tree 코어, Role B는 인덱스/파일 일관성, Role C는 SQL 입력/검증과 API 계약/HTTP 메시지, Role D는 실행/서버 런타임을 맡고, 테스트와 문서는 각 역할 책임에 포함시킨다.

이 구조로 가면 기존 소유권 규칙을 깨지 않으면서도, API 서버와 스레드 풀 같은 신규 구현을 균형 있게 병렬 진행할 수 있다.
