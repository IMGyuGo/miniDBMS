# miniDBMS 4인 역할분담 및 진행 계획

## 1. 문서 목적

이 문서는 현재 저장소 상태를 기준으로, 앞으로 `miniDBMS` 프로젝트를 4명이 병렬로 진행할 때 충돌을 줄이고 속도를 높이기 위한 역할분담 초안이다.

나중에 `AGENT.md`를 만들 때 이 문서를 기준으로 파일 소유권, 공통 규칙, 작업 순서를 확정하는 것을 목표로 한다.

---

## 2. 현재 저장소 상태 요약

현재 코드 기준으로 확인한 내용은 아래와 같다.

| 구분 | 현재 상태 |
|------|-----------|
| 엔진 진입점 | `src/main.c` 에 CLI 실행 흐름이 있다. |
| SQL 처리 | `src/input`, `src/parser`, `src/schema`, `src/executor` 가 연결되어 있다. |
| 인덱스 | `src/index/index_manager.c`, `src/bptree/bptree.c` 가 존재하고 executor와 연결되어 있다. |
| 지원 기능 | `INSERT`, `SELECT`, `WHERE col = value`, `WHERE col BETWEEN a AND b` 가 들어가 있다. |
| 스키마 | 현재 `schema/users.schema` 기준으로 `users` 테이블을 사용한다. |
| 데이터 저장 | `data/{table}.dat` 파일 기반이다. |
| 서버 계층 | `src/server`, `src/http`, `src/threadpool` 같은 API 서버 계층은 아직 없다. |
| 네트워크/동시성 | HTTP 처리, 소켓 처리, 요청 큐, 스레드 풀, 락 구조는 아직 없다. |
| 테스트 | `Makefile`에는 테스트 타깃이 있지만 `tests` 폴더는 비어 있다. |
| 문서 | `docs` 폴더는 비어 있었고, `README.md`도 아직 비어 있다. |

즉 지금 상태는 "CLI 기반 DB 엔진은 이미 있고, 앞으로 API 서버와 동시성 계층, 테스트, 문서를 붙여야 하는 단계"로 보는 것이 가장 맞다.

---

## 3. 역할 분담 원칙

역할을 나눌 때 가장 중요한 기준은 아래 3가지다.

1. 이미 있는 엔진 코드를 최대한 재사용한다.
2. 파일 충돌이 많이 나는 구간과 새로 만드는 구간을 분리한다.
3. 서버 계층은 엔진 내부를 직접 건드리지 않고, 중간 서비스 계약을 통해 붙인다.

이 기준으로 보면 가장 안정적인 분담은 "엔진 입력/실행", "저장/인덱스", "서버/동시성", "테스트/문서"의 4축이다.

## 4. 권장 4인 역할분담

| 역할 | 담당 영역 | 주 소유 파일/디렉터리 | 핵심 목표 | 바로 시작할 일 |
|------|-----------|------------------------|-----------|----------------|
| Role 1 | SQL 입력, 파싱, 검증, 실행 진입부 | `src/input/**`, `src/parser/**`, `src/schema/**`, `src/executor/**`, `src/main.c` | CLI 엔진을 서버에서도 재사용 가능한 형태로 정리 | SQL 실행 공용 함수 경계 정리, 에러/결과 구조 정리, CLI 유지 |
| Role 2 | 저장/인덱스/B+Tree | `src/index/**`, `src/bptree/**`, `include/index_manager.h`, `include/bptree.h` | 데이터 파일과 인덱스 경로를 안정화하고 성능/복구 기반 마련 | `index_init`, `db_insert`, range 조회 경로 점검, 재시작 복구와 예외 케이스 정리 |
| Role 3 | API 서버, HTTP, 스레드 풀, 동시성 | 신규 `src/server/**`, `src/http/**`, `src/threadpool/**`, 필요 시 신규 `include/server*.h` | 외부 클라이언트가 SQL을 네트워크로 호출할 수 있게 만들기 | `POST /query` 스펙 정리, 요청 큐, 워커 스레드, read/write lock 구조 설계 |
| Role 4 | 테스트, 벤치마크, README, docs, 데모 | `tests/**`, `docs/**`, `samples/**`, `README.md`, 필요 시 `scripts/**` | 팀 결과물을 검증 가능하고 발표 가능한 상태로 만들기 | 테스트 구조 생성, curl 예제, 시나리오 문서, README 초안 작성 |

## 5. 각 역할의 상세 책임

### Role 1. SQL 입력/실행 계층 담당

이 역할은 현재 가장 많이 연결된 기존 엔진 경로를 맡는다.

담당 이유:
- 이미 `lexer -> parser -> schema -> executor -> main` 흐름이 이어져 있다.
- API 서버가 붙더라도 결국 SQL 문자열을 받아 같은 엔진으로 흘려보내야 한다.
- 따라서 서버와 엔진 사이에 재사용 가능한 실행 진입점을 만드는 사람이 필요하다.

주요 책임:
- CLI 전용 흐름을 서버에서도 호출 가능한 공용 실행 흐름으로 정리
- SQL 에러 메시지와 실행 결과를 구조화
- `SELECT`와 `INSERT` 실행 결과를 서버 응답에 싣기 좋게 다듬기
- 필요하면 다중 statement 처리 정책 정리

권장 산출물:
- 엔진 공용 진입 함수 설계안
- SQL 실행 결과 구조체 초안
- parser/schema/executor 테스트 초안
- CLI와 서버가 함께 쓸 수 있는 호출 예시

### Role 2. 저장/인덱스/B+Tree 담당

이 역할은 DB 엔진의 성능과 안정성 핵심을 맡는다.

담당 이유:
- API 서버가 붙으면 요청 수가 늘어나면서 파일/인덱스 안정성이 더 중요해진다.
- 현재 프로젝트의 차별점이 B+Tree와 인덱스 사용 경로이기 때문에 발표에서도 중요하다.
- 동시성 제어를 넣기 전에도 먼저 "무엇을 보호해야 하는가"를 명확히 해야 한다.

주요 책임:
- `data/{table}.dat` 와 인덱스의 일관성 유지
- 서버 재시작 시 인덱스 재구성 전략 정리
- 중복 키, 잘못된 row, 빈 파일, 대량 데이터 등의 예외 처리 정리
- 성능 비교에 필요한 측정 포인트 유지

권장 산출물:
- 인덱스 초기화/복구 흐름 문서
- insert/select 경로에서 락이 필요한 지점 정리
- B+Tree 및 index 단위 테스트 초안
- 성능 측정 기준 정리

### Role 3. API 서버/동시성 담당

이 역할은 이번 과제의 신규 구현 본체를 맡는다.

담당 이유:
- 현재 저장소에는 서버 계층이 아예 없다.
- 새로 생기는 코드가 많아서 기존 엔진 담당과 분리해야 충돌이 적다.
- 스레드 풀, 큐, 소켓, HTTP, 락은 하나의 흐름으로 묶여 있어 한 역할이 소유하는 편이 낫다.

주요 책임:
- 최소 HTTP API 설계
- 요청 큐와 고정 크기 워커 스레드 풀 구현
- DB 엔진 호출 전후의 락 정책 설계
- 에러 응답, 큐 포화, 종료 처리, 타임아웃 처리

권장 1차 목표:
- `POST /query` 하나로 시작
- 요청 body 에 SQL 문자열 전달
- 응답은 `rows`, `elapsed_ms`, `error` 중심의 단순 JSON
- 초기 버전은 global mutex 또는 rwlock 기반으로 안전성 우선

권장 산출물:
- 서버 폴더 구조
- 요청 처리 흐름도
- 스레드 풀 API
- 최소 동작 가능한 API 서버 MVP

### Role 4. 테스트/문서/발표 담당

이 역할은 "만든 기능을 증명하는 팀의 바깥면"을 맡는다.

담당 이유:
- 현재 `tests` 폴더가 비어 있고 `README.md`도 비어 있어서 결과물을 설명하고 검증할 층이 필요하다.
- 발표는 결국 동작, 테스트, 문서, 데모 품질로 평가된다.
- 다른 역할이 개발에 집중할 수 있도록 검증/정리/시연 자료를 분리하는 것이 효율적이다.

주요 책임:
- 테스트 파일 구조 생성
- 단위 테스트, 통합 테스트, API 테스트, 동시성 테스트 정리
- README와 docs 작성
- curl/Postman 예시, 데모 시나리오, 발표용 검증 흐름 정리

권장 산출물:
- `tests/test_parser.c`, `tests/test_schema.c`, `tests/test_executor.c`, `tests/test_index.c`, `tests/test_bptree.c` 초안
- 이후 `tests/test_api.c`, `tests/test_threadpool.c`, `tests/test_concurrency.c` 추가
- README 초안
- 발표용 실행 시나리오 문서

## 6. 파일 소유 범위 제안

나중에 `AGENT.md`로 옮기기 쉽게, 파일 소유 범위는 아래처럼 정리하는 것을 권장한다.

| 역할 | 우선 소유 범위 |
|------|----------------|
| Role 1 | `src/input/**`, `src/parser/**`, `src/schema/**`, `src/executor/**`, `src/main.c` |
| Role 2 | `src/index/**`, `src/bptree/**`, `include/index_manager.h`, `include/bptree.h` |
| Role 3 | `src/server/**`, `src/http/**`, `src/threadpool/**`, 신규 서버 관련 헤더 |
| Role 4 | `tests/**`, `docs/**`, `samples/**`, `README.md`, 테스트/문서 목적의 `scripts/**` |

공통 합의 파일 후보:

- `include/interface.h`
- `Makefile`
- 새로 추가될 공용 헤더 (`include/db_service.h`, `include/server_api.h` 같은 파일)

공통 합의 파일은 한 명이 독단 수정하지 말고, 변경 전에 함수 시그니처와 구조체를 먼저 합의하는 방식이 좋다.

## 7. 실제 진행 순서 추천

### 1단계. 공용 계약 먼저 고정

이 단계에서는 각자 구현보다 "붙는 면"을 먼저 맞춰야 한다.

- Role 1: 엔진 공용 호출 방식 초안 작성
- Role 2: 인덱스/파일 일관성 체크 포인트 정리
- Role 3: 서버 요청/응답 스펙 초안 작성
- Role 4: 테스트 매트릭스와 README 목차 초안 작성

이 단계가 끝나면 최소한 아래 3가지는 정해져 있어야 한다.

- 서버가 엔진을 어떤 함수로 호출할지
- 응답을 어떤 구조로 받을지
- 동시성 제어를 어디에 걸지

### 2단계. 병렬 구현

- Role 1: CLI 경로를 공용 서비스 호출로 정리
- Role 2: index/bptree 안정화, 복구/예외 처리 보강
- Role 3: HTTP 서버, 큐, 워커 스레드 구현
- Role 4: 단위 테스트 뼈대와 API 테스트 문서 작성

### 3단계. 통합

- Role 1 + Role 3: 서버에서 엔진 호출 연결
- Role 2 + Role 3: 락 범위와 파일/인덱스 일관성 점검
- Role 4: 전체 시나리오 테스트와 문서 갱신

### 4단계. 발표 준비

- Role 4가 README와 데모 흐름 정리
- Role 2가 성능 비교 포인트 정리
- Role 3이 API 동작 데모 준비
- Role 1이 SQL 처리 흐름 설명 자료 정리

## 8. 역할 간 인터페이스 규칙

작업 충돌을 줄이려면 아래 규칙을 미리 두는 것이 좋다.

1. Role 3은 엔진 내부 구현을 직접 호출하지 않고, Role 1이 정한 공용 진입 함수를 통해서만 호출한다.
2. Role 2는 락을 직접 서버 코드에 흩뿌리지 말고, 어떤 구간이 보호 대상인지 먼저 문서화한다.
3. Role 4는 테스트를 "나중에" 하지 말고 각 역할의 완료 기준에 바로 연결한다.
4. `include/` 아래 공용 헤더 변경은 짧게라도 4명 합의 후 반영한다.
5. `Makefile` 수정은 마지막에 몰아서 하지 말고, 새 타깃이 생길 때 바로 반영한다.

---

## 9. 이 분담안의 장점

이 구조로 나누면 장점이 분명하다.

- Role 1과 Role 2는 기존 엔진 내부를 안정화한다.
- Role 3은 새 서버 계층을 독립적으로 빠르게 만들 수 있다.
- Role 4는 테스트와 문서 품질을 계속 끌어올릴 수 있다.
- 공용 파일 충돌이 `include/`와 `Makefile` 정도로 줄어든다.
- 발표 준비가 마지막에 몰리지 않는다.

---

## 10. 최종 제안 요약

현재 저장소 기준으로 가장 현실적인 4인 분담은 아래 한 줄로 정리된다.

> Role 1은 SQL 입력/실행 계층, Role 2는 저장/인덱스 계층, Role 3은 API 서버/동시성 계층, Role 4는 테스트/문서/발표 계층을 맡는다.

이 구조로 가면 기존 CLI 엔진을 버리지 않고 재사용할 수 있고, 이번 과제 핵심인 API 서버와 스레드 풀도 병렬로 진행할 수 있다.
