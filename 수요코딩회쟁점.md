# 🧠 미니 DBMS - API 서버 과제 정리

## 📌 과제 본질

이 과제는 기존의 **CLI 기반 미니 DBMS를 “외부 프로그램이 네트워크로 호출할 수 있는 서버”로 확장**하는 작업

핵심:

- UI를 만드는 것이 아니라
- 기존 SQL 처리기 + B+ 트리를 **서버 뒤에 붙이고**
- 여러 요청을 **안전하게 처리하는 구조를 만드는 것**

## 🏗️ 전체 아키텍처

```

외부 클라이언트 (curl / Postman / 웹앱 / 모바일 앱)
↓
API 서버 (HTTP 추천)
↓
요청 큐 + 스레드 풀
↓
DB 엔진 (lexer / parser / schema / executor / index / b+tree)
↓
JSON 응답 또는 에러 응답

```

## 1️⃣ 외부 클라이언트에서 DBMS 기능 사용

👉 의미:

- UI를 반드시 만들라는 뜻 ❌
- **네트워크로 DB 기능을 호출 가능하게 하라는 뜻**

### ✔️ 가장 현실적인 구현

- HTTP API 서버
- `POST /query` 하나로 시작

```
POST /query
Content-Type: application/json

{
    "sql": "SELECT * FROM users;"
}
```

응답 예시:

```json
{
    "rows": [...],
    "elapsed_ms": 3
}
```

### ✔️ 클라이언트 예시

- curl
- Postman
- 웹앱 / 모바일 앱

👉 발표/demo는 curl로도 충분

---

## 2️⃣ 스레드 풀(Thread Pool)

### ✔️ 개념

> 요청마다 스레드를 새로 만드는 대신
> 👉 **미리 만들어둔 스레드를 재사용하는 구조**

---

### ✔️ 동작 흐름

1. 메인 스레드가 요청을 받음
2. 요청을 큐에 넣음
3. 워커 스레드가 큐에서 꺼냄
4. SQL 실행
5. 응답 반환
6. 스레드는 종료되지 않고 재사용

---

### ✔️ 사용하는 이유

- 스레드 생성 비용 절감
- 과도한 스레드 증가 방지
- 자원 사용 제어 가능

---

### 💡 한 줄 정의

> 스레드 풀 = “병렬 처리용 인력 풀”

---

## 3️⃣ 멀티 스레드 구조

👉 웹 서버 구조와 거의 동일

### ✔️ 기본 구조

```
accept() 스레드
↓
job queue
↓
worker threads
↓
DB 실행
↓
응답 반환
```

---

## ⚠️ 진짜 어려운 부분: 동시성

👉 네트워크보다 DB 동기화가 더 어려움

### 문제

- 파일 동시 접근
- 인덱스 갱신 충돌
- 전역 상태 공유 (Lock 필수)

---

### ✔️ 현실적인 선택지

#### 1. MVP (X)

- global mutex
- 모든 SQL 직렬 실행

#### 2. 개선 버전 (실제로 개발)

- rwlock
  - SELECT → read lock (동시 실행 가능)
  - INSERT → write lock (단독 실행)

👉 발표용으로 가장 적합

---

## 4️⃣ API 서버 ↔ DB 엔진 연결

👉 핵심: **레이어 분리**

---

### ❌ 나쁜 구조

- HTTP 코드 + DB 코드 섞임

---

### ✅ 좋은 구조

```
[API 계층]

- HTTP 처리
- JSON 파싱
- 응답 생성

    ↓

[서비스 계층]

- execute_sql(sql, result)

    ↓

[DB 엔진]

- lexer
- parser
- executor
- index
- b+tree
```

---

### ⚠️ 중요한 변화

기존 CLI:

```

printf()

```

API 서버:

```

구조화된 결과 반환 (JSON)

```

👉 출력 로직 분리 필수

## 🔥 주요 쟁점

### 1. API 프로토콜

- HTTP + JSON (추천)

---

### 2. 엔진 완성도

- parser / executor OK
- bptree / index_manager → 병목 가능

---

### 3. 출력 → 응답 구조 변경

- printf → JSON 구조 필요
  - 즉, API 서버는 rows, error, elapsed_ms 같은 구조화된 응답 필요

---

### 4. 동시성 제어

- 동시에 INSERT 작동시 파일 오프셋과 인덱스 갱신 순서 충돌 위험

---

### 5. 병렬성 수준

- 네트워크 병렬 vs DB 병렬 구분 필요
- 스레드 풀은 있어도 DB 전체 mutex를 걸면 SQL 실행은 직렬일 수 있음

---

### 6. 서버 재시작

- `.dat` 파일 기반 인덱스 복구 필요
- 서버 재시작 시 인덱스를 복구 해야지 서버 재시작 후에도 검색 가능

---

### 7. 에러 처리

- 잘못된 SQL
- 테이블 없음
- 중복 키
- 큐 포화
- 클라이언트 중도 종료

---

### 8. 테스트

- 단위 테스트
- API 테스트
- 동시성 테스트
- 엣지 케이스 테스트
