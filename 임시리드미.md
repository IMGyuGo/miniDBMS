# 임시 리드미

이 문서는 이 프로젝트를 처음 보는 사람이 전체 구조를 빠르게 이해할 수 있도록 만든 임시 안내서다.

## 1. 이 프로젝트는 무엇인가

이 프로젝트는 C로 만든 작은 SQL 실행기다.

- 지원 중심 기능: `INSERT`, `SELECT`
- 저장 방식: `data/{table}.dat` 텍스트 파일에 한 줄씩 저장
- 스키마 정의: `schema/{table}.schema`
- 인덱스: B+Tree 2개
  - `id` 인덱스
  - `age` 인덱스
- 목적:
  - SQL 실행 흐름을 직접 구현해 보기
  - 선형 탐색과 B+Tree 인덱스 탐색의 차이를 비교해 보기

즉, "파일 기반 간단한 DB + B+Tree 인덱스 실험"이라고 생각하면 된다.

## 2. 큰 실행 흐름

사용자가 `sqlp`에 SQL 파일을 넘기면 대략 아래 순서로 동작한다.

1. SQL 파일을 읽는다.
2. Lexer가 토큰으로 쪼갠다.
3. Parser가 AST를 만든다.
4. Schema가 이 SQL이 유효한지 검사한다.
5. Executor가 실제 `INSERT` 또는 `SELECT`를 수행한다.
6. 필요하면 Index Manager를 통해 B+Tree를 사용한다.

흐름을 한 줄로 쓰면 아래와 같다.

```text
SQL file
 -> lexer
 -> parser
 -> schema validation
 -> executor
 -> index manager
 -> bptree
 -> data file
```

## 3. 디렉터리별 역할

  같은 구조를 해석한다.

주요 파일:

- `src/parser/parser.c`

### `src/schema/`

- 테이블 스키마를 읽는다.
- SQL이 스키마와 맞는지 검사한다.
- 컬럼 존재 여부, 타입 적합성 등을 본다.

주요 파일:

### `src/executor/`

- 실제 실행 담당
- `INSERT`면 파일에 쓰고 인덱스를 갱신한다.

주요 파일:

- `src/executor/executor.c`

주요 파일:

## 4. 실제 데이터는 어떻게 저장되나

이 프로젝트는 DB 파일 포맷이 복잡하지 않다.

- 파일 위치: `data/{table}.dat`
- 한 줄이 한 row
- 컬럼 구분자: `" | "`

예시:

```text
1 | user_0000001 | 18 | user1@example.com
2 | user_0000002 | 19 | user2@example.com
```

즉, `INSERT`는 사실상 이 파일 뒤에 한 줄을 append 하는 작업이다.

## 5. INSERT는 어떻게 동작하나

`INSERT`의 핵심은 "파일에 쓰고, 그 줄의 시작 위치(offset)를 인덱스에 넣는다"는 점이다.

순서는 아래와 같다.

1. `data/{table}.dat`를 `"ab"` 모드로 연다.
2. 현재 파일 끝 위치를 `ftell()`로 구한다.
3. row를 한 줄로 기록한다.
4. 그 offset을 `id` / `age` 인덱스에 넣는다.

왜 offset이 중요하냐면, 나중에 `SELECT`에서 인덱스가 이 값을 돌려주고 executor가 그 위치로 바로 jump 해서 row를 읽기 때문이다.

## 6. SELECT는 어떻게 동작하나

현재 executor는 쿼리 모양을 보고 경로를 정한다.

### 인덱스 경로를 타는 경우

- `WHERE id = ?`
  - `index_search_id`
- `WHERE id BETWEEN ? AND ?`
  - `index_range_id_alloc`
- `WHERE age BETWEEN ? AND ?`
  - `index_range_age_alloc`

### linear 경로로 가는 경우

- `WHERE name = 'alice'`
- `WHERE email = '...'`
- 그 외 인덱스 규칙에 맞지 않는 조건
- 또는 `--force-linear` 옵션 사용 시

### 주의할 점

인덱스는 "row의 offset을 빨리 찾는 것"까지가 핵심 역할이다.

즉 아래 두 단계로 나뉜다.

1. B+Tree가 offset을 찾는다.
2. executor가 그 offset으로 실제 row를 다시 읽는다.

그래서 결과 row가 많으면, 인덱스 탐색은 빨라도 실제 row fetch는 느릴 수 있다.

이 점 때문에 `age BETWEEN 30 AND 40` 같은 쿼리는 linear보다 느릴 수도 있다.

## 7. 이 프로젝트의 인덱스 구조

테이블 하나당 인덱스가 두 개 있다.

- `id` 트리
  - key: `id`
  - value: file offset
- `age` 트리
  - key: `age`
  - value: file offset

`age`는 중복값이 많기 때문에 같은 key에 여러 offset이 매달릴 수 있다.

## 8. compare 모드는 무엇인가

`--compare`는 같은 SELECT를 두 번 실행한다.

- auto path
- forced linear path

stdout에는 아래 순서로 결과 테이블이 나온다.

```text
[AUTO RESULT]
...
[LINEAR RESULT]
...
```

stderr에는 요약 한 줄이 나온다.

예시:

```text
[COMPARE] auto=index:id:eq/0.114ms/3 linear=linear/2.815ms/0 speedup=24.69x
```

마지막 숫자는 `tree_io`다.

## 9. `tree_h`와 `tree_io`는 무엇인가

### `tree_h`

- 트리 높이
- 루트부터 leaf까지 몇 단계인지 보여준다.

### `tree_io`

- 이 프로젝트에서 "노드 페이지 몇 번 방문했는가"를 보여주는 지표
- sim 빌드에서는 각 노드 방문마다 page-read 시뮬레이션도 수행한다.

즉:

- `tree_h`는 구조적 깊이
- `tree_io`는 실제 탐색 중 방문 수

라고 보면 된다.

## 10. `test_perf.c`는 무엇을 비교하나

`test_perf.c`는 두 층위 비교를 한다.

### 1. Raw 비교

- 인덱스 API 자체
- 직접 구현한 선형 파일 스캔

여기서는 ResultSet을 만들지 않고, 순수 lookup/filter 비용을 본다.

### 2. Executor 비교

- 실제 `db_select()` 인덱스 경로
- 강제 linear 경로

여기서는 사용자가 실제로 느끼는 비용에 더 가깝다.

즉 `Raw`는 "트리 자체가 빠른가", `Executor`는 "실제 최종 실행이 빠른가"를 본다.

## 11. 주요 테스트 항목

현재 성능 벤치는 크게 4개를 본다.

### Test 1. Point Search

```sql
SELECT * FROM users WHERE id = ...
```

- id point lookup
- 인덱스가 가장 잘 먹히는 대표 예시

### Test 2. Range Search (id)

```sql
SELECT * FROM users WHERE id BETWEEN ... AND ...
```

- id range query
- leaf 순회가 어떻게 동작하는지 보기 좋다.

### Test 3. Range Search (age)

```sql
SELECT * FROM users WHERE age BETWEEN 30 AND 40
```

- secondary index range query
- 결과 row가 많을 때 왜 linear가 더 빠를 수 있는지 드러나는 구간

### Test 4. Height Comparison

- `order=4`와 `order=128` 비교
- 트리 높이 차이를 일부러 과장해서 본다.

이 테스트는 "실서비스 튜닝"보다는 "높이가 늘면 어떤 일이 생기는가"를 보기 위한 실험용에 가깝다.

## 12. 자주 헷갈리는 포인트

### 1. 인덱스가 항상 더 빠른 것은 아니다

아니다.

- 결과가 적으면 인덱스가 유리
- 결과가 너무 많으면 linear가 유리할 수 있음

특히 `age BETWEEN`처럼 많은 row가 걸리면, offset을 많이 따라가야 해서 executor 단계에서 느려질 수 있다.

### 2. `tree_io`가 작다고 전체가 무조건 빠른 것은 아니다

아니다.

B+Tree는 offset을 빨리 찾았더라도, 그 뒤에 실제 row를 다시 읽는 비용이 클 수 있다.

### 3. `Raw` 결과와 `Executor` 결과는 달라도 된다

그렇다.

- `Raw`: 트리/선형 필터 자체만 비교
- `Executor`: row fetch + parse + ResultSet 생성까지 포함

그래서 `Raw index`가 빨라도 `Executor index`는 느릴 수 있다.
